MMR CLI · Design Vision

From code reviewer to second-opinion engine

MMR already does the hard part — fan one question out to many independent models and reconcile where they agree. Today it only ever points that engine at a diff. Point it at a design, and the same machine answers the question every builder actually asks: is this the best way to solve this — and what did we miss?

00 The one command that should exist

You design a feature with Claude Code. It hands you a solution. A good one — but is it the best one? What would a different model have done? The honest answer today is a manual ritual: open other chat tools, re-paste the context into each, read three replies, and synthesize them in your head. MMR was built to kill exactly that ritual — for code. It should do it for designs too.

Today — the manual second opinion
$ # Claude Code proposes a design...
$ # open ChatGPT, paste the problem + the proposal
$ # open Gemini, paste it all again
$ # read 3 replies in 3 tabs, dedupe by hand,
$ # guess which concerns are real because
$ # two models independently raised them
The vision — one command
$ mmr critique design.md --context repo
CRITIQUE · design.md · 4 models · advisory
●●●● polling won't scale past ~10k users  [consensus]
▲    queue vs cron — models split        [divergence]
SYNTHESIS push-based delivery is the
         single strongest change.

The thesis

MMR's real asset isn't bug-finding — it's independent multi-model fan-out plus reconciliation: ask N models the same thing without letting them see each other, then surface where they converge. That machine is artifact-agnostic. It is pointed at diffs only because its prompt, output shape, and verdict were written for code. Give it a second framing and a second output shape and the same engine becomes a design/brainstorm second-opinion tool — the place independent models are worth the most, because the cost of the wrong approach is highest before you build it.

01 What MMR is today — grounded in the code

A code review flows through six stages. Only three of them carry any assumption that the artifact is code — and all three are swappable.

1 · Resolve input
resolveDiff() turns --pr/--staged/--diff/--base..head into one unified-diff string (review.ts:37–99). Code-specific.
2 · Build prompt
assemblePrompt() layers a core template (the "find bugs / security / correctness" framing) + criteria + focus + the diff (prompt.ts:42 · templates/core-prompt.md). Code-specific.
3 · Dispatch
Each channel runs in isolation — subprocess or HTTP, stdin or prompt-file — in parallel, never seeing another's output (dispatcher.ts · http-dispatcher.ts). Artifact-agnostic.
4 · Parse
Each reply is parsed into Finding{ severity, location, category, description, suggestion } (parser.ts · types.ts:45). Shape is code-specific.
5 · Reconcile
reconcile() groups semantically-similar findings across channels and labels each consensus | majority | unique with a confidence (reconciler.ts:43–168). Artifact-agnostic — and exactly the signal a design review needs.
6 · Verdict
evaluateGate() + deriveVerdict() turn findings into pass / degraded-pass / blocked against a P0–P3 threshold (reconciler.ts:229–256). Code-specific.

The load-bearing insight

Stages 3 and 5 — dispatch and reconciliation, the genuinely hard, valuable parts — already don't care what the artifact is. Reconciliation even computes consensus / majority / unique, which is precisely "where do independent reviewers agree vs. diverge." A design critique is not a new engine. It is a new prompt (stage 2), a new output shape (stage 4), and no gate (stage 6) — bolted onto the engine that already exists.

02 The gap — second opinions are worth the most at design time

Code review catches defects in an approach you've already committed to. A design review questions the approach itself — when changing it is still cheap.

  • The expensive mistakes are architectural, not lexical. A missed edge case costs a patch. A wrong data model or a doomed scaling strategy costs a rewrite. Those decisions are made in the design, reviewed by no one, and shipped to an agent that is confident by construction.
  • One model is one perspective. Claude Code is excellent and also singular. The thing it didn't consider, it cannot tell you it didn't consider. Independent models with different priors are the cheapest way to find the blind spot.
  • Convergence is the signal you can't get alone. When four models that never saw each other's answers independently flag the same risk, that's a strong prior something is wrong. When they split, that's a genuine judgment call you should make consciously. MMR's reconciler already measures both — it's simply never been pointed at a design.
  • The manual workflow is the tax. Re-pasting context into three chat tools and reconciling by hand is exactly the friction MMR removed for diffs. Designs deserve the same one-command treatment.

03 First principles — a second-opinion engine

Repositioning MMR means naming what a multi-model second-opinion tool is for, then back-casting onto the existing engine. Eight principles fall out.

SO1

Independence is the product

Models must never see each other's answers. The whole value is uncorrelated perspectives; cross-contamination collapses four opinions into one. This is already enforced for review — preserve it exactly.

SO2

Advisory, never authority

A critique informs a human decision; it does not gate, block, or fail. Always exit 0. The output's job is to make you think, not to approve or reject your design.

SO3

Convergence vs. divergence is the headline

Where independent models agree = high signal. Where they split = your judgment call. This distinction, not a flat list, is the most decision-useful thing a multi-model tool can show — and the reconciler already computes it.

SO4

Meet the artifact where it is

A design doc, a pasted "problem + proposed solution", an implementation plan. Normalize them internally; never demand a rigid format. The friction of preparing input is the friction that kills adoption.

SO5

Critique generatively, not just critically

The value isn't only "what's wrong" — it's "what else could you do" and "what did you not consider." Surface alternatives and missed considerations, not just flaws. A second opinion expands the option space.

SO6

Ground in reality on request

"Is this the best approach?" is sharper when the models can see the actual system. Let a critique optionally pull in codebase + docs, so feedback judges fit against your architecture — not design-in-a-vacuum.

SO7

One engine, two kinds

Code review and design critique share dispatch and reconciliation; only framing, output shape, and verdict differ. Reuse the primitives; don't fork the tool. Repositioning is additive, not a rewrite.

SO8

Iterate cheaply

Design is iterative. A critique should refine across rounds — you revise, re-run, and the models see what changed and what they said last time. Lean on MMR's existing session machinery, not a new stateful service.

04 The new verb — mmr critique

A distinct, discoverable command that signals "this isn't a code review." It reuses the channel engine underneath, but its grammar, prompt, and output are its own. (Final name is an open question — §10 OQ4.)

Flexible input — three ways in

# 1 · a design doc / spec / plan on disk
$ mmr critique docs/design/notifications.md

# 2 · a pasted "problem + proposed solution" over stdin
$ mmr critique - <<'EOF'
PROBLEM: notify users when a job finishes.
PROPOSED (by Claude Code): poll a status table every 30s from the client.
EOF

# 3 · an implementation plan, with the real codebase in view
$ mmr critique docs/plans/2026-07-01-billing.md --context repo

The surface

--context repo
Optionally ground the critique in the codebase + docs so models judge fit against your real system. Default: artifact-only.
--session <id>
Continue an iterative critique — models see prior rounds and your revisions (§6).
--focus "…"
Free-text steer ("focus on scaling", "is the data model right?") — same mechanism as review's focus.
--channels / --format
Same channel selection and text | json | markdown output controls as mmr review. JSON is first-class for agents.

Why a verb, not a flag

Repositioning MMR as a general second-opinion engine means the design mode should be as discoverable as review — a first-class verb an agent or human can find in --help and the capability manifest, not a mode buried behind review --mode design. The engine is shared; the identity is distinct.

05 What a critique produces

Not a verdict. A structured, advisory report whose spine is consensus-vs-divergence, ending in a synthesis. Severity (P0–P3) and pass/blocked don't fit open-ended design — they're replaced by kind and agreement.

$ mmr critique docs/design/notifications.md --context repo

CRITIQUE · notifications.md · 4 models · advisory (no gate)

CONVERGENCE  — independent models agreed (high signal)
  ●●●● polling every 30s won't scale past ~10k users; 3 of 4
       urge push/SSE or webhooks instead.        [concern · consensus]
  ●●●○ no story for delivery failure / retries.    [missed · majority]

DIVERGENCE  — models split (your judgment call)
  ▲ Codex + Claude: add a durable queue.
    Antigravity: a queue is premature for v1 — cron sweep.  [alternative · split]

ALTERNATIVES the proposal didn't consider
  • Server-Sent Events over a dedicated channel        (Claude, Codex)
  • Reuse the existing job store, not a new table       (local-ai)

OPEN QUESTIONS to resolve before building
  • What is the real concurrency target? 3 models flagged it unstated.

SYNTHESIS
  Sound for a prototype; polling is the main risk at scale. The single
  strongest change is push-based delivery. Resolve the concurrency
  target first — it decides queue-vs-cron.

The shape under it

A code-review Finding becomes a CritiqueItem: kind (concern | alternative | consideration | open-question) instead of severity; a free theme instead of a file:line location; and the same sources[] + agreement the reconciler already produces. No verdict, no fix_threshold — a critique never blocks. A final synthesis (one model reading the reconciled set) turns the map into a recommendation. (Item taxonomy and the synthesis pass are open questions — §10.)

06 Iterative refinement — session-based

Design is a loop. The critique should be too — without a new stateful service. Round one opens a session; you revise; the next run feeds the models the prior rounds and your changes.

$ mmr critique design.md
  → opens session s-7f3a; round 1 critique above

$ # ...you rewrite design.md to use SSE + a concurrency target...

$ mmr critique design.md --session s-7f3a
  ROUND 2 — models see round 1's concerns + your diff
  ✓ scaling concern resolved — SSE addresses it (4/4 agree)
  ✓ concurrency target now stated
  ▲ new: SSE reconnect/back-pressure unaddressed     [concern · majority]

Built on MMR's existing sessions store — each round is a fresh, independent fan-out, but the prompt carries a compacted memory of prior rounds. How that memory is summarized to stay bounded is an open question (§10 OQ7).

07 Repo-grounded critique — judging fit, not abstractions

"Is this the best approach?" is far sharper when the reviewer can see what you're building on. --context repo bundles relevant codebase + docs into the critique so feedback is about your system, not a textbook.

  • Default off. Artifact-only is the simplest, most private, and right for greenfield/abstract questions. Opt in when fit matters.
  • It changes the answers. "Reuse the existing job store, not a new table" is only possible if the model can see the job store. Grounding turns generic advice into specific advice.
  • It costs tokens. The whole repo won't fit and shouldn't. How to select context — whole-tree digest vs. relevant-file heuristic vs. user-specified paths — is an open question (§10 OQ3).

08 Architecture — three ways to build it

The engine is shared; the question is how cleanly the two kinds coexist without destabilizing the code-review path that thousands of reviews depend on.

A

Mode parameter on the existing engine

Thread mode: 'code' | 'design' through assemblePrompt / reconcile / deriveVerdict / formatters; critique is a thin command that sets it. Maximum reuse — but sprinkles conditionals through the stable review core.

B · recommended

Parallel pipeline on shared primitives

critique is its own command with its own prompt, parser, and output module, reusing only the stable primitives — the channel dispatcher and the reconciler. Code review is untouched; the design path can evolve freely. A little duplication, bought with isolation.

C

Generalize the core into "review kinds"

Refactor so code and design both implement a small ReviewKind interface (prompt, parser, output, verdict). The most elegant expression of "one engine, two kinds" — and the biggest blast radius on a stable path. The right eventual shape, the wrong first step.

→

The call

Build B: a parallel critique pipeline reusing the dispatcher + reconciler, with the shared primitives factored cleanly so it reads as C without the risk. Code review never changes behavior. If a second non-code kind ever appears, the C refactor is then obvious and safe.

Backward-compatibility guarantee

mmr review, its config, its findings shape, its verdict semantics, and the channel/auth/job-store machinery are untouched. critique is purely additive. The only shared, evolving code is the dispatcher and reconciler — both already artifact-agnostic.

09 Roadmap — value first, depth after

Ordered so the first phase delivers a usable second opinion on its own, and each later phase deepens it. Exact flags and edge cases get pinned per-phase in implementation plans. (Phase boundaries may shift after the §10 consult.)

Phase 1 · Core critique the usable MVP

A one-shot, artifact-only design critique that already beats the manual ritual.

  • New mmr critique verb; flexible input (file · stdin · problem+solution)
  • templates/critique-prompt.md — design framing (alternatives, missed considerations, risks, prior art)
  • CritiqueItem / CritiqueReport types; reuse the reconciler for agreement
  • Advisory output (text + json), no gate, always exit 0

Phase 2 · Convergence + synthesis the differentiator

Make the multi-model nature pay off explicitly.

  • Render the convergence / divergence map as the headline
  • A synthesis pass — one model reads the reconciled set and recommends
  • Surface model disagreement as a first-class section, not noise

Phase 3 · Repo-grounded judge fit, not abstractions

Let the critique see the real system.

  • --context repo — bundle relevant codebase + docs into the critique
  • Context-selection strategy (resolved via §10 OQ3)

Phase 4 · Iterate + teach close the loop

Multi-round refinement and the docs that make it discoverable.

  • Session-based iterative refinement (--session); bounded round memory
  • Config defaults for critique; mmr commands + explain entries
  • Teach it: CLAUDE.md, the mmr skill, the reference guide

10 Open questions — to resolve by multi-model consult

Before planning, each of these goes independently to Codex, Claude, Antigravity, and local-ai-delegate; answers get consolidated and folded back into this roadmap. (This section becomes "Decisions — resolved" after the consult.)

OQ1 · Item taxonomy
Are concern / alternative / consideration / open-question the right kinds? Should items carry an optional importance, or stay unranked to avoid smuggling a severity gate back in?
OQ2 · Representing disagreement
What's the most decision-useful way to show models in genuine conflict (A says X, B says ¬X)? A diff-style split? A spectrum? This is the differentiator — get it right.
OQ3 · Repo-context selection
For --context repo: whole-tree digest, a relevant-file heuristic, agent-supplied paths, or a hybrid? How to stay within token budgets without starving the critique.
OQ4 · Naming
critique vs. advise vs. second-opinion vs. design-review — and the conceptual noun for the unified engine (does review stay the umbrella, or does a new noun sit above both?).
OQ5 · Same prompt vs. personas
Send every channel the same critique prompt (rely on model diversity), or assign distinct lenses (skeptic / simplifier / user-advocate / domain-expert) to maximize perspective spread? Does persona assignment compromise independence?
OQ6 · Synthesis pass
Should a final model synthesize a recommendation, or is that over-reach for an advisory tool? If yes — which model, and how to keep it from laundering one model's opinion as "the answer"?
OQ7 · Bounded iteration memory
Across rounds, how is prior-round context compacted so the prompt doesn't grow without bound while still letting models judge "did my revision address your concern"?

11 Non-goals (YAGNI)

What this deliberately is not, to keep the surface honest and the engine stable.

  • Not a replacement for human judgment. A critique is input to your decision, never the decision. It never auto-applies a suggestion or rewrites your design.
  • Not a gate. Design critique never blocks a merge or fails CI by default. It is advisory by construction (a configurable soft-gate was explicitly declined).
  • Not a fork of code review. mmr review keeps its exact behavior. We add a kind; we don't rebuild the tool.
  • Not a code generator. It evaluates and expands an approach; it doesn't produce the implementation.
  • No new long-running service. Iteration rides the existing session store; no daemon, no watch mode.
