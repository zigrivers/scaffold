MMR CLI · Design Vision

Making MMR legible to the agents that drive it

An AI agent spent 20 minutes and 8 commands trying to turn off one review channel. The capability already existed — the agent just couldn't see it. This is a vision for a CLI an agent can read, mutate, and self-heal in seconds.

00 The one command that should have existed

A trivial goal — disable a channel that fails on every review — took an exploratory expedition. The fix the agent ultimately hand-wrote into a YAML file in a path the CLI never names should have been a single, guessable command.

Today — ~20 min, 8 commands
$ mmr config
action  [choices: "init","test","channels"]   # read-only
$ mmr config channels show:grok
# ...read docs site for precedence...
# ...discover ~/.mmr/config.yaml is never mentioned...
$ nano ~/.mmr/config.yaml
channels:
  grok:
    enabled: false   # hand-edited, unguided
The vision — 5 seconds, 1 command
$ mmr config disable grok
✓ Disabled channel 'grok'
  wrote   ./.mmr.yaml  (channels.grok.enabled: false)
  now     grok  disabled  (project .mmr.yaml)
  revert  mmr config enable grok

The thesis

MMR is read-rich and write-poor, and its richest signals are invisible at the moment an agent needs them. Almost every capability the agent wanted already exists in the code — disabling channels, provenance tracking, install/auth diagnostics, auto-skip. The gap is not capability. It is legibility: can an agent discover, mutate, and verify state without spelunking? This vision closes that gap, starting from what a perfectly agent-legible CLI would feel like and back-casting to concrete MMR changes.

01 What happened — grounded in the code

The agent's feedback was sharp and mostly correct. Reading the actual source (MMR v1.7.0) sharpens it further — and reveals that the real story is more "hidden" than "missing."

The trigger
grok failed/timed out on all 11 reviews in a session. Each failure spawned a compensating retry — added latency, zero findings. Root cause: grok wasn't installed.
The goal
Turn the channel off. The agent expected mmr channels disable grok. There is no such command.
The cost
~8 exploratory commands, a separate docs site, and a hand-edited YAML file in ~/.mmr/config.yaml — a path the CLI never prints.

What the code actually shows

  • Write-poor is real. mmr config accepts exactly init | test | channels (config.ts:337), and channels is read-only. There is no enable, disable, or set.
  • But the capability already exists — twice. The config schema supports two ways to turn a channel off: channels_disabled: ["grok"] and per-channel enabled: false. Both require hand-editing YAML. The missing piece is a mutator, not a model.
  • Provenance already exists — but hides. loadConfigWithProvenance tracks every value's source and mmr config channels show <name> prints per-field # from <source> (config.ts:208–230). Yet the list view shows no source, there's no whole-config view, and no command prints where config files live — even though the loader computes both paths (loader.ts:302, :143).
  • The compensating-pass waste is the real bug. Review does auto-skip uninstalled channels (review.ts:550–556) — then fires a compensating pass for them every run (review.ts:725+). That's the "added latency, zero findings" the agent felt. Nothing tells the agent "you don't have grok → disable it."
  • The diagnostic is already there, just unsurfaced. mmr config test reports "grok": { installed: false } with a recovery hint. The tool knew the whole time — it just never said so at the point of pain, nor suggested running it.
  • The "docs say init|show|validate" claim is a red herring. The README, reference guide, and skill all correctly document init|test|channels. The real documentation gap is that the agent-facing skill teaches nothing about disabling a channel, and --help ships no examples.

The pattern under all six points

MMR gives an agent excellent read access and almost no write path, then hides its best diagnostics one command away from where they're needed. An agent reasons from what a tool says — --help, output, errors. If the help doesn't list it and the error doesn't teach it, the agent concludes it doesn't exist and starts spelunking.

02 First principles — what a perfectly agent-legible CLI feels like

Before fixing MMR, design the ideal. Agents use a CLI differently from humans: every exploratory command costs context; they infer capability from text; they carry strong priors from the thousands of CLIs in their training; and they act decisively on explicit, machine-readable instructions. Ten principles fall out of that.

P1

Every capability self-discloses

If the tool can do it, --help and a machine-readable manifest list it with a runnable example. An agent should never have to guess a command exists. Undiscoverable = nonexistent.

P2

Read and write are symmetric

If you can show a value, you can set it — in the same namespace. The universal agent loop is inspect → mutate → re-inspect. No read-only islands.

P3

State is locatable & attributed

Every effective value traces to its source file and layer. A path command discloses the search order and which files exist. No invisible config surfaces.

P4

Errors are remediations

Every failure prints the one command that fixes it. The diagnostic the tool already computes is surfaced at the point of pain, not hidden behind a command the agent must know to run.

P5

Self-heal what's safe; propose the rest

A structurally-dead channel auto-skips with one remediation line — it does not silently retry-then-compensate-then-fail every run. Distinguish "transiently down" from "permanently absent."

P6

Machine-readable is a first-class mode

Every human view has a --json twin emitting the same data — value, source, remediation — so agents parse deterministically instead of scraping prose.

P7

Convention over novelty

Match the noun-verb-object grammar agents already know (config set key value). Avoid bespoke syntax like show:codex that no agent has priors for.

P8

One obvious path; aliases for the rest

The 90% task has a single guessable command. Synonyms are fine as fallbacks, but --help leads with the canonical one.

P9

Teach just-in-time

The lesson lives in the output at the moment of need, not only in a manual the agent must know to open. Context-local teaching beats a separate guide.

P10

Mutations are safe & self-confirming

Every mutator is idempotent, prints what changed and where it wrote, and shows how to revert. An agent never fears running it.

The rest of this document back-casts these principles onto concrete MMR features. Each feature is tagged with the principles it serves and the phase it lands in.

03 Mutate config — give the agent a write path

The headline gap. MMR can read everything and change nothing through the CLI. Three commands close it, all built on config machinery that already exists.

A1 mmr config disable / enable <channel>

Phase 1P2P7P8P10

Problem. The 90% task — "turn this channel off" — has no command. The agent must learn the YAML schema and pick one of two undocumented mechanisms. config.ts:337 · schema supports channels_disabled + enabled:false

Proposed. A dedicated, idempotent verb. Writes the canonical per-channel enabled: false (D5) using a comment-preserving YAML writer (D2) to the project .mmr.yaml by default — but a disable targeting a not-installed channel routes to global with a notice (D1). --global/--project override. Prints the write location, the new effective value with provenance, and the revert command; enable also prunes a stale channels_disabled entry (D5).

Before
$ mmr config disable grok
Unknown argument: disable
# agent falls back to editing YAML by hand
After
$ mmr config disable grok
✓ Disabled channel 'grok'
  wrote   ./.mmr.yaml
  now     grok  disabled  (project .mmr.yaml)
  revert  mmr config enable grok

A2 mmr config set / unset <dotted.path> <value>

Phase 2P2P7P10

Problem. Disabling is one of many edits — thresholds, timeouts, model overrides — that all currently mean hand-editing YAML. There's no general mutator, and MMR's own ecosystem already standardizes one: Scaffold's cli-conventions.md documents tool config set key value and explicitly recommends noun-verb-object. MMR violates its own house convention.

Proposed. A general dotted-path setter with a comment-preserving YAML writer, mirroring bd config set (already used across Scaffold). unset removes an override and prints the inherited value it falls back to.

Before
# edit .mmr.yaml by hand:
defaults:
  fix_threshold: P1
channels:
  codex:
    timeout: 600
After
$ mmr config set defaults.fix_threshold P1
✓ defaults.fix_threshold: P2 → P1  (./.mmr.yaml)
$ mmr config set channels.codex.timeout 600
✓ channels.codex.timeout: 300 → 600  (./.mmr.yaml)
$ mmr config unset channels.codex.timeout
✓ unset — reverts to default (300)

Design decision — where do writes land?

Default: project .mmr.yaml (the change travels with the repo and the team sees it). --global writes ~/.mmr/config.yaml for machine-wide preferences like "I never have grok installed." Every mutator prints which file it touched (P3/P10) so the choice is never silent.

04 See config — make state locatable and attributed

The provenance engine already exists; it's just under-exposed. Surface it everywhere, add a path command, and give every view a --json twin.

B1 mmr config path

Phase 1P3P1

Problem. The CLI never discloses where config is read from or written to. The agent had to find the precedence rule in a separate docs site. loader.ts computes both paths at :302 and :143

Before
$ mmr config path
Unknown argument: path
After
$ mmr config path
Search order (later wins):
  1 built-in defaults      (always)
  2 ~/.mmr/config.yaml     ✓ exists
  3 ./.mmr.yaml            ✗ not found
  4 CLI flags              (per-invocation)
write target (default): ./.mmr.yaml
                  --global → ~/.mmr/config.yaml

B2 Provenance in the list view + mmr config show (whole config)

Phase 1P3P6

Problem. Per-field provenance only appears in single-channel show <name>. The list view shows name/enabled/command/parser with no source, and there's no effective-whole-config view. So "is grok off because of a default or my file?" needs a guess.

Before — mmr config channels
[
  { "name": "grok", "enabled": false, ... },
  { "name": "claude", "enabled": true, ... }
]
# no source — can't tell default from override
After — mmr config channels
CHANNEL    STATUS    SOURCE
claude     enabled   default
codex      enabled   default
grok       disabled  ~/.mmr/config.yaml
gemini     disabled  default
# --json emits {name,enabled,source,file} per row

Proposed. Annotate every value with its source (extending the existing provenance plumbing to the list + a new whole-config mmr config show). Add --json to all of them so an agent reads {value, source, file} structurally rather than scraping (P6).

05 Self-healing channels — stop dispatching the dead ones

The most impactful behavior change. MMR should classify why a channel is unavailable and act accordingly — heal the transient, skip-and-remediate the structural, and never silently burn latency on a channel you'll never have.

C1 Classify unavailability: transient vs. structural

Phase 2P5P4

Problem. Review treats every unavailable channel the same: skip, then fire a compensating pass — every run. For a channel that's merely re-auth-able that's correct; for one that isn't installed and never will be, it's pure waste. review.ts:550 (skip) · :725+ (compensate)

Proposed. The status is already computed (not_installed vs auth_failed vs disabled). Branch on it:

auth_failed transient
Compensate (it'll come back after re-auth) and print the recovery command.
not_installed structural
Skip with a one-line remediation. Do not compensate by default — opt in with --compensate-missing or mark the channel required: true.
disabled intentional
Silent. The agent already chose this.
Before — every review
$ mmr review --pr 47
...
grok: failed (0s)
[compensating: Grok-equivalent] dispatched…
# +latency, 0 findings, repeats forever
After
$ mmr review --pr 47
...
grok: skipped — not installed (structural)
      not compensating a channel you don't have
      → mmr config disable grok  (silence this)

Back-compat note → resolved as D3

This flips a default for a published tool, so the consolidated release ships as major (mmr 2.0.0) with a "Behavior changes" CHANGELOG entry, a one-time runtime notice, and the --compensate-missing / required: true opt-outs. The CLAUDE.md guidance ("run a compensating pass for each missing external channel") is updated to distinguish transient from structural — see §07 (E3) and the decision in §11.

C2 mmr doctor — diagnose and remediate in one shot

Phase 2P4P5P9

Problem. mmr config test already knows install + auth status, but emits raw JSON with no remediation plan, and nothing points the agent to it. The diagnostic and the fix live in different places.

Proposed. One command that fuses them: classify every channel, print a human plan with the exact remediation per channel, and offer --fix to apply the safe ones (disable structurally-absent channels). --json for agents.

Before — mmr config test
{
  "grok":   { "installed": false, "auth": "skipped" },
  "gemini": { "installed": true,  "auth": "failed" }
}
# now what? no plan, no next step
After — mmr doctor
Channels: 3 ready · 1 needs auth · 1 absent

 ✓ claude   installed, authed
 ✓ codex    installed, authed
 ⚠ gemini   auth expired  → gemini -p "hello"
 ✗ grok     not installed → install, or:
                              mmr config disable grok

Run `mmr doctor --fix` to disable absent channels.

C3 Point-of-pain surfacing

Phase 1P4P9

Problem. When a review degrades, the summary says grok: failed (0s) — a dead end. The tool never suggests mmr config test or a fix.

Proposed. Every degraded channel in a review summary carries its remediation inline and a pointer to mmr doctor. The lesson arrives where the pain is (P9), one parse away from action (P4). This is a Phase-1 win because the statuses already exist — it's a formatting change.

06 Discover & use — let an agent load the whole surface at once

Beyond config: an agent should be able to learn what MMR can do in one call, not by probing N commands and accumulating context. This is where "agent-native" goes furthest.

D1 Rich --help with runnable examples

Phase 1P1P8

Problem. --help lists choices but no examples, and leads with the bespoke show:<channel> colon form. An agent guessing from help can't tell the canonical path from a fallback.

Proposed. Add yargs .example() lines that lead with canonical commands (mmr config disable grok, mmr config channels show codex). Keep show:codex as a back-compat alias — but the help leads with the conventional positional form (P7/P8).

D2 Machine-readable capability manifest — mmr commands --json

Phase 3P1P6

Problem. To learn MMR's surface, an agent runs many --help calls and scrapes prose — expensive in context and lossy.

Proposed. One command emits the entire command tree as structured data — each command, its args, flags, and a runnable example. An agent (or an MCP wrapper) loads the whole capability surface in a single deterministic call. The CLI becomes a self-describing API (P1/P6).

$ mmr commands --json
[
  { "command": "config disable <channel>",
    "summary": "Turn a review channel off",
    "example": "mmr config disable grok",
    "writes": true, "idempotent": true },
  { "command": "doctor",
    "summary": "Diagnose channels and print fixes",
    "example": "mmr doctor --fix", "writes": false },
  ...
]

D3 mmr explain <topic> — just-in-time docs from the CLI

Phase 3P9

Problem. The good guidance lives in a guide the agent must already know exists. Discovery requires prior knowledge — the opposite of legible.

Proposed. mmr explain channels, mmr explain config, mmr explain compensation print the relevant concept inline. The CLI carries its own teaching surface, so an agent that hits a wall can ask the tool itself instead of guessing a docs URL (P9).

07 The Scaffold side — teach agents the new surface

A better MMR CLI only helps if the agent-facing material Scaffold ships actually teaches it. The 20-minute spelunk happened partly because the mmr skill — the thing agents actually read — never mentions disabling a channel.

E1 · Phase 1

mmr SKILL.md — add "Configuring channels"

The highest-leverage doc fix. content/skills/mmr/SKILL.md teaches init/test/channels but nothing about turning a channel off. Add a section covering disable/enable/set, mmr doctor, and where config lives. This is what agents load at the point of need.

E2 · Phase 1

Reference guide + knowledge

Surface config-mutation prominently in content/guides/mmr/index.md (not buried in a config appendix) and regenerate index.html (drift gate). Mention doctor/disable in content/knowledge/core/multi-model-review-dispatch.md pre-flight guidance.

E3 · Phase 2

Wrapper behavior + CLAUDE.md

review-pr.md / review-code.md and the CLAUDE.md "compensating pass for each missing external channel" rule should adopt the transient-vs-structural distinction (§05 C1): re-auth transient channels, skip-and-remediate structural ones — don't over-compensate.

E4 · Phase 1

Internal-consistency win

Scaffold's own content/knowledge/cli/cli-conventions.md already prescribes noun-verb-object (config set key value). Citing it makes A1/A2 a consistency fix, not a new opinion — MMR conforming to the house standard it ships.

Net Scaffold verdict

No Scaffold CLI changes are required for Phase 1 — the wrappers call MMR and inherit its improvements. The required Scaffold work is content: teach the new commands in the skill/guide/knowledge so agents discover them, and (Phase 2) align the compensating-pass guidance with the new classify behavior.

08 Roadmap — surface first, heal second, self-describe third

Ordered by leverage-per-effort. Phase 1 is mostly surfacing machinery that already exists — high impact, low risk. Phase 2 adds genuine behavior change. Phase 3 is the agent-native frontier.

Phase 1 · Quick wins days · surfacing existing machinery

Turn the 20-minute spelunk into a 5-second command, mostly by exposing what's already computed.

  • A1 config enable/disable <channel> — the headline command (introduces the comment-preserving yaml writer, D2; not-installed→global routing, D1)
  • B1 config path — disclose the search order & write target
  • B2 provenance + source in the list view; --json twins
  • C3 point-of-pain remediation lines in review output
  • D1 --help examples; lead with the canonical show <channel>
  • E1, E2, E4 teach the new surface in the mmr skill, guide, knowledge

Phase 2 · Self-healing ~1–2 weeks · behavior change

Stop wasting latency on dead channels; give the agent a one-shot diagnose-and-fix.

  • A2 config set/unset general setter (comment-preserving writer)
  • C1 classify transient vs. structural; flip the compensating-pass default → consolidated release is major, mmr 2.0.0 (D3)
  • C2 mmr doctor (+ --fix)
  • E3 align wrapper + CLAUDE.md compensation guidance

Phase 3 · Agent-native exploratory · self-describing CLI

Let an agent (or an MCP layer) consume MMR as a structured, self-teaching API.

  • D2 mmr commands --json capability manifest
  • D3 mmr explain <topic> inline docs
  • required vs. optional channel semantics so compensation is always intentional
  • optional: an MCP/JSON contract over the whole config + review surface

10 Non-goals (YAGNI)

What this vision deliberately does not build, to keep the surface small and the changes safe.

  • No interactive TUI / GUI config editor. Agents drive a CLI; a visual editor adds surface no agent uses.
  • No config daemon or watch mode. Config is read on invocation; that's the right model. Don't add a long-running process.
  • No new config schema. Every mutation targets fields that already exist. We're adding write paths and visibility, not new concepts.
  • No auto-install of missing CLIs. doctor --fix disables absent channels; it does not silently install third-party tools (security + consent).
  • No change to the merge/precedence model. The four-layer deep-merge stays exactly as-is; we expose it, we don't redesign it.

11 Decisions — resolved by multi-model consult

Each open question was put independently to three models — Codex, Claude, and Antigravity — and consolidated. Consensus was strong; the only split was on Q3's semver, resolved below. These decisions are now binding inputs to the implementation plans.

D1 · Write target — RESOLVED (all 3 → option c)

Mutators default to the project ./.mmr.yaml; --global/--project always override and the chosen path is always printed. Special case: config disable <channel> for a channel whose CLI is not installed defaults to global ~/.mmr/config.yaml with a one-line notice ("grok CLI not found — recording as a machine-level preference; pass --project to scope to this repo"), since absence is a property of the laptop, not the repo. mmr doctor flags the stale-disable case (a globally-disabled channel whose CLI later appears).

D2 · Comment preservation — RESOLVED (all 3 → eemeli yaml)

Use the yaml (eemeli) Document API — parseDocument() → setIn()/deleteIn() → toString() — to edit values in place, preserving comments, key order, and scalar style. Adds yaml as a write-side dependency alongside the existing js-yaml reader. Guard four pitfalls: schema-aware type coercion (CLI strings → typed bool/number scalars, never "false"), anchors/aliases/merge keys, multi-document files (refuse with a clear error), and missing intermediate keys. Fixture tests cover comments, anchors, arrays, and missing paths.

D3 · Compensation back-compat — RESOLVED (flip default; ship as MAJOR)

Flip the default: do not compensate structurally-absent (not_installed) channels; continue compensating transient (auth_failed) ones. Two of three models (Codex, Antigravity) classed the default-flip as a major change; Claude argued minor-but-loud. Consolidated: since the product value is the better default, flip it and ship the consolidated release as MAJOR (mmr 2.0.0). Escape hatches preserve old behavior: --compensate-missing and per-channel required: true. Mitigations: a "Behavior changes" CHANGELOG heading, a one-time runtime notice the first time a structurally-absent channel is skipped-without-compensation (naming the channel + the opt-outs), and the Scaffold rule update (E3). --json changes stay additive (new status/reason/remediation fields).

D4 · Redaction — RESOLVED (all 3 → single boundary, default-on)

Redact once, over the in-memory config model, before any serialization boundary — so text, --json, effective-config, and the capability manifest all inherit it by construction. Default-on; the only bypass is --no-redact, which prints a loud warning to stderr so it can never contaminate stdout JSON. Guard four failure modes: recurse into nested maps/arrays; positional KEY value token pairs (["--header","Authorization: Bearer …"]); name vs. value (keep api_key_env: OPENAI_API_KEY names, redact api_key values); and unknown/future fields via a value-pattern/entropy scan (sk-, ghp_) so new secret fields fail closed. Extends the existing redact.ts guards rather than reinventing them.

D5 · Canonical disable — RESOLVED (all 3 → per-channel enabled: false)

The mutator writes per-channel channels.<id>.enabled: false (round-trips cleanly with config set, gives precise provenance, deep-merges as a predictable scalar). channels_disabled: [...] stays honored on read as legacy (it is implemented today — review.ts:500–503). Precedence: an explicit per-channel enabled wins over channels_disabled membership. config enable <ch> also prunes a stale channels_disabled entry at the target layer (and warns if it lives in a layer it can't edit) so enable can't silently fail. doctor --fix offers to migrate legacy lists to per-channel form.

12 What I recommend next

If this direction is right, the natural next step is to turn Phase 1 into an implementation plan — it's high-impact, low-risk, and mostly surfaces machinery that already exists. The headline command (config disable/enable) plus point-of-pain remediation and the skill update would have prevented the entire 20-minute spelunk that prompted this.

Recommendation

Approve the vision (or redirect it), then I'll invoke the planning skill to write a Phase-1 implementation plan: TDD'd config enable/disable + config path + provenance list view + review remediation lines, paired with the mmr skill/guide updates. Phases 2–3 each get their own plan when we're ready.

This document is a vision, not a spec — it commits to a direction and a command surface, but exact flags, output formats, and edge cases get pinned down per-phase in the implementation plans.
