Build Observability

A durable record of what agents decide, claim, and get blocked on during a build — fused with git, GitHub, MMR, and pipeline state, then audited against the planning docs by nine lenses. This page explains the why and the how, citing the code that is the ground truth.

01

The problem

Why Build Observability exists

When several agents (and humans) build a project in parallel worktrees, the reasoning behind the build evaporates. A decision gets made in a throwaway branch, a blocker is hit and worked around, a story is silently dropped — and none of it survives the squash-merge. The docs say one thing; the code drifts to another. Build Observability turns those ephemeral moments into a durable, queryable record, then audits the record against the planning docs.

What goes wrong without it

  • Decisions die at teardown. An agent records "we switched the cache to Redis" in its head, opens a PR, the worktree is removed — and the rationale is gone. The next agent re-litigates it.
  • Blockers go unaddressed. A dependency block is hit at hour 0 and nobody notices it is still open at hour 6.
  • Phase boundaries pass unaudited. tech-stack.md is marked complete while it still contradicts the PRD; the contradiction is found three phases later.
  • Doc/code drift accumulates. A component is used outside its declared layer; an acceptance criterion never gets a test; a coding-standard erodes file by file.

Failure mode — events lost across a worktree teardown

The harvester (src/observability/engine/harvester.ts) and scripts/teardown-agent-worktree.sh exist precisely to close this gap — see §11.

Architecture

System map

One write path, one fusion point, two read paths. Agents write events to the append-only ledger; adapters synthesize events from git, GitHub, MMR and pipeline state; the synthesizer fuses them with correlation-id dedup; the engine API drives the audit lenses and the progress timeline; renderers emit terminal, markdown, dashboard and MMR-findings output. Click any box for its file and a jump to the relevant section.

Boxes labelled probe only contribute availability checks (and, for audit_history, trend data) but do not emit events into the replay timeline — see §4.

03

The write layer

The ledger

Every durable observation is one JSON object on one line of .scaffold/activity.jsonl. Writes are append-only and lock-guarded so parallel worktrees never corrupt the file; each event is capped at 4 KiB; and secrets and home-directory paths are scrubbed both when the event is written and again when output is rendered.

How a write happens

  • Validate against the per-type payload allow-list (event-schemas.ts). Unknown payload keys are dropped (reported in dropped_fields); a bad ts or missing required field rejects the event.
  • Redact the payload (redact.ts) — see below.
  • Size-check: Buffer.byteLength(line, 'utf8') must be ≤ 4096, else the write throws (ledger-writer.ts:10,58).
  • Append under a proper-lockfile lock (10 retries, exponential backoff) so concurrent agents serialize cleanly.

The nine event types

Each event carries a common envelope — event_id (ULID), worktree_id, actor_label, branch, task_id (string or null), type, ts (ISO-8601 UTC) — plus a type-specific payload. The CLI verb is scaffold observe event <type> … (§6).

Worktree identity

Each worktree gets a stable identity on first write — .scaffold/identity.json holds worktree_id (a UUID), worktree_label (derived from the directory name, or primary), and created_at (identity.ts). The id is what lets the harvester tell one worktree's events from another's, and what stamps every event's worktree_id.

External signals

Adapters

The ledger only holds what agents chose to record. Adapters fill in the rest by synthesizing events from the surrounding tools — commits, PRs, MMR jobs, pipeline state, test runs. There are eight adapters (AdapterId at types.ts:69), but only five emit events into the timeline; the other three are availability probes / trend helpers.

Source priority

When two events describe the same thing (same correlation_id), the synthesizer keeps the one from the most authoritative source. The order is fixed in SOURCE_PRIORITY at synthesizer.ts:254 — lower number wins, ties broken by earliest ts:

1mmrauthoritative build verdict
2ghGitHub PR state of record
3gitlocal commit history
4statepipeline step status
5testscached test run

The three probe-only adapters are intentionally absent from this table — they never produce a replay event to dedup. The fusion algorithm itself is in §7.

05

The audit

The nine-lens audit

The audit reads the planning docs (PRD, stories, tech-stack, standards, design system, plan, playbook) into a document graph, then runs a set of independent lenses over the graph, the code, the ledger and adapter data. The design spec calls it the "eight-lens audit" (A–H); a ninth lens — I-knowledge-gaps — joined the suite after v3.26.0.

8 in the spec, 9 in the code …build-observability-design.md titles §3 "The Eight Audit Lenses" and its LENS_REGISTRY stops at H. The shipped registry.ts:26 adds I-knowledge-gaps. Treat Lens I as the ninth lens that joined the suite — documented here as a first-class member.

Scope, order and skips

  • Scope → lenses (api.ts:72): --scope code runs A–G; --scope docs runs H, I; --scope all runs all nine. An explicit --lens <id> overrides the scope selection.
  • Registry order: A → B → C → D → E → F → G → H → I. G-decisions declares depends_on: ['D-stack'] so it runs after D and can consume D's unsanctioned-dependency findings.
  • Missing adapter → skip: if a lens's required adapters are unavailable it is skipped and emits a P3 finding with evidence.kind = lens_skipped. Skipped lenses (not blocking findings) are what turn a pass into a degraded-pass.

Lens by lens

A-tdd

TDD violations

Are there skipped tests? — lens-a-tdd.ts

Reads: the doc-graph's discovered tests (with last_status), ACs, stories, and ac_to_test edges. Tunes: none. Severity: P0 skipped test on a must-priority story; P1 otherwise.

Example: src/tests/auth.test.ts — skipped test "verify email OTP" → P0; fix hint add_test ("Re-enable test…").

B-ac-coverage

AC completion

Is every acceptance criterion covered by a passing test? — lens-b-ac-coverage.ts

Reads: ACs, tests, ac_to_test edges; optional tests adapter for live status. Tunes: none. Severity: P0 AC's test is failing; P1 AC has no test edge or unknown status.

Example: AC-001 has no ac_to_test edge → P1 (ac_not_covered).

C-standards

Coding-standards drift

Are coding standards being violated across files? — lens-c-standards.ts

Reads: graph rules + files. Tunes: lenses.C-standards.rule_overrides (per-rule severity), escalation_threshold (default 5 — more than N violations of a rule escalates to P1), enforce_via_linter. Severity: rule-specific P0–P3.

Example: rule no-console-log violated at src/index.ts:42, 8 occurrences → escalated P1.

D-stack

Tech-stack drift

Unsanctioned dependencies, or components used outside their layer? — lens-d-stack.ts

Reads: file_to_component_use edges, components (with layer), and decision_recorded ledger events (which can sanction a dependency). Tunes: lenses.D-stack.path_to_layer (default 7 glob→layer mappings). Severity: P0 unsanctioned dependency; P1 out-of-layer use.

Example: src/utils/auth.ts imports @unsafe-crypto with no covering decision → P0; fix hint record_decision.

E-design

Design-system drift

Are ad-hoc design values bypassing tokens? — lens-e-design.ts

Reads: file_to_token_use edges, tokens (with priority + category). Tunes: lenses.E-design.ad_hoc_token_threshold (default 3), ui_glob (default src/components/**/*.tsx, consumed by the doc-graph builder). Severity: P1 ad-hoc count over threshold; P0 must-priority token category bypassed.

F-scope

Missing scope

Is high-priority work missing stories/plans, or are planned stories untouched? — lens-f-scope.ts

Reads: features, stories, plan-tasks and their edges; task_claimed ledger events; optional state adapter. Tunes: lenses.F-scope.untouched_story_grace_hours (default 168 = 7 days). Severity: P0 must-feature without story/plan; P1 should-feature; P2 planned-but-untouched story past grace.

G-decisions

Undocumented decisions

Decisions made in the ledger/commits but not documented (or vice versa)? — lens-g-decisions.ts

Reads: graph decisions, decision_recorded events, recent git commits (keyword scan), and D-stack's findings. Tunes: none effective (see note). Severity: P0 unsanctioned dep from D with no decision; P1 ledger↔doc mismatch; P2 decision-keyword commit with no event/doc.

Inert override The spec advertises a lenses.G-decisions.keywords_file override and ships src/observability/checks/data/decision-keywords.txt, but loadKeywords() hard-codes its keyword list and never reads the config key or the file. The override and the bundled file are currently inert — flagged in §16.
H-cross-doc

Cross-doc inconsistency

Are features, stories, plans, playbook and decisions internally consistent? — lens-h-cross-doc.ts

Reads: the full structural doc-graph + unresolved globs. Severity: P0 decision supersedes a non-existent decision; P1 feature with no story / orphan story / uncovered story; P2 plan task not in playbook, unresolved glob.

Lens H — LLM-graded full-profile sub-checks Under --profile=full only, Lens H runs three extra checks via an LLM dispatcher (all silently return [] if the dispatcher fails, so the audit never breaks):
  • tech-stack ↔ PRD — contradictions (P0) or tensions (P2) between PRD prose and the chosen stack.
  • PRD → stories coverage — a prose feature with no covering story (P1).
  • terminology drift — one concept named inconsistently across docs (P2).
Security — the LLM dispatcher command is hard-coded The Lens H dispatcher (llm-dispatcher.ts) always invokes claude -p through a fixed shell; it is deliberately not project-config-overridable, because the audit may run against untrusted third-party repos and a repo-controlled command would be arbitrary code execution. Only llm.timeout_s (and llm.parallel_checks) are configurable. Contrast the --fix dispatcher, which is configurable — see §10.
I-knowledge-gaps

Knowledge gaps — the ninth lens

What topics have agents repeatedly needed (over a 90-day window) with no knowledge-base entry? — lens-i-knowledge-gaps.ts

Reads: knowledge_gap_signal ledger events + synthetic signals scanned from tasks/lessons.md (lens-i-lessons-scanner.ts). Severity: P1 ≥5 signals from ≥3 projects; P2 ≥3 signals from ≥2 projects.

Lens I — --knowledge-root and existing-entry suppression The lens resolves the knowledge directory in three tiers (knowledge-index.ts): the --knowledge-root CLI flag → lenses.I-knowledge-gaps.knowledge_root in config → auto-detect by walking up for content/knowledge/. If a gap's topic slug already exists in the resolved knowledge index, the finding is suppressed (no P1/P2). If no root resolves, suppression is disabled and the lens warns once per audit. Shipped in #397 (lens + lessons scanner) and #406 (--knowledge-root + suppression).

Verdict taxonomy

The engine computes exactly three verdicts (Verdict at types.ts:6):

verdictwhenexit
passno blocking findings, no skipped lenses0
degraded-passno blocking findings, but ≥1 lens skipped (missing required adapter)0
blocked≥1 open finding at or above fix_threshold1
"needs-user-decision" is not an engine verdict The code review workflow has a needs-user-decision outcome, but that is an MMR review verdict, not one the audit engine emits. The audit engine's universe is exactly the three above.

fix_threshold

A finding is blocking when its status is open and its severity is at or above the threshold (fix-threshold.ts). Resolution order: --fix-threshold flag → .mmr.yaml (audit_fix_threshold / fix_threshold) → default P2. The threshold never hides findings — every finding is still reported; it only decides which ones count toward blocked.

07

CLI

Progress, replay & stall

scaffold observe progress shows what's in flight. With --replay it fuses the ledger with synthesized git/gh/mmr/state/tests events into one timeline; stall detection runs at every command boundary and surfaces a "Needs Attention" panel.

Fusion — how the timeline is built

The synthesizer (synthesizer.ts:254) converts each ledger Event to a ReplayEvent, concatenates the adapter events, filters to the time window, then dedups:

  • Events with a null correlation_id pass through untouched.
  • Events that share a correlation_id collapse to one — the synthesizer keeps the lowest SOURCE_PRIORITY number (ledger 0 → tests 5; see §4), breaking ties by earliest ts.
  • The surviving events sort by (ts, source-priority, sort_id).

So a PR that the agent recorded (pr_opened in the ledger) and that the gh adapter also reports collapses to the single ledger event, because ledger (0) outranks gh (2).

Stall detection — the six signals

At every command boundary the engine checks for staleness and builds needs_attention (stall.ts). The type declares six signals; five fire today — pr_review_stale is reserved (deferred pending a per-PR correlation_id on the mmr adapter, stall.ts:119). Automatic audits

Phase-boundary triggers

Completing a planning document should re-check the docs for drift — automatically, but never blocking the build. When a pipeline step at a phase boundary is marked complete, the state manager fires a docs audit and prints a one-line summary.

The hook

StateManager.markCompleted() is async (state-manager.ts:174): it saves state, then — if the completed step is a phase boundary — awaits runPhaseAudit({ primaryRoot, step }). The six boundary steps (phase-subsets.ts):

user-storiestech-stackcoding-standards design-systemimplementation-planimplementation-playbook

The audit is fixed to H-cross-doc, scope=docs, profile=fast (phase-audit.ts). It is non-gating: markCompleted catches any audit error and returns a result object, so a state transition never fails because of the audit.

The doc-conformance channel is disabled by default. Enable it per-project in .mmr.yaml or for a single run with --channels=doc-conformance; if it's globally enabled you can opt a project out via channels_disabled: ["doc-conformance"].

Remediation

The --fix flow

scaffold observe audit --fix doesn't just report blocking findings — it dispatches an agent to fix each one, verifies the fix with a single-lens re-audit, and writes a post-fix report. The loop is abort-safe: an interrupt restores the working tree to where it started.

phase 1
audit
collect findings + verdict
phase 2
buildFixPlan
keep open & ≥ threshold; order by severity then lens
phase 3
dispatchFixAgent
subprocess, prompt on stdin
phase 4
verify
re-run that one lens; retry ≤ 3

docs/audits/<id>-postfix.{md,json}

The five phases

  • 1 · audit — run the audit to get findings and the verdict (fix-flow.ts).
  • 2 · buildFixPlan (fix-plan.ts) — keep findings that are open and at/above fix_threshold, ordered by severity ascending (P0 first) then lens_id.
  • 3 · dispatchFixAgent (fix-agent-dispatcher.ts) — spawn the dispatcher command, write the per-finding prompt to stdin, stream stdout/stderr to the user.
  • 4 · single-lens verify — re-run only the finding's lens; if the finding is gone, mark fixed; else retry, up to per_finding_max_attempts (default 3).
  • 5 · post-fix report — a final scope=all audit, written to docs/audits/<id>-postfix.{md,json}.

Abort safety

Before dispatching, captureSnapshot (abort-snapshot.ts) records a git stash create SHA plus the set of pre-existing staged files. Newly staged paths are tracked per attempt. On SIGINT, restoreSnapshot un-stages what the loop staged and re-applies the stash — so an interrupted fix leaves a clean, WIP-preserving tree.

The fix dispatcher IS configurable — unlike Lens H's fix.dispatcher_command is read from project config because the fix loop only runs against the maintainer's own repo, not untrusted third-party code (fix-agent-dispatcher.ts). This is the opposite of the Lens H LLM dispatcher (§5), which hard-codes its command for untrusted-repo safety. CLAUDE.md's --fix note describing the dispatcher as "not project-config-overridable" conflates the two — flagged in §16.

Exit code: 1 if any blocking finding survived the loop, else 0.

11

Lifecycle

Harvest, recover & teardown

A worktree's ledger is local to that worktree. Harvest copies it into the primary repo's archive before the worktree is removed; recover sweeps up ledgers whose worktrees vanished without being harvested; teardown does the whole dance in one command.

Commands

  • scaffold observe harvest --worktree=<path> — read the worktree's identity.json, then copy its activity.jsonl to .scaffold/activity-archive/active/<worktree_id>.jsonl in the primary repo (atomic temp-file + rename under a lock). Requires the primary root to be a real repo, not itself a worktree.
  • scaffold observe harvest --recover — list live worktrees (git worktree list --porcelain), read each one's worktree_id, then scan active/ for entries whose worktree is no longer live and rotate each into activity-archive/<YYYY-MM>.jsonl (bucketed by the file's mtime), deleting the active entry. Returns the rotated ids.
  • scripts/teardown-agent-worktree.sh <path> — derives the primary repo via git -C <path> rev-parse --git-common-dir (so it works from anywhere), harvests (fail-soft), runs git worktree remove, then deletes the workspace branch if it differs from the primary HEAD.
Empty ledgers are fine If a worktree never wrote an activity.jsonl, harvest is a clean no-op and teardown proceeds — exactly the case recorded for the five observability workspaces in §15.
This page is not the teardown audit The worktree-by-worktree teardown audit lives separately at docs/superpowers/observability-worktree-status.md (on the feat/observability-worktree-audit-report branch as of this writing). This section documents the mechanism; that doc records a specific run.

Configuration

Config reference

All tunables live in .scaffold/observability.yaml. The file is optional — when absent, DEFAULT_CONFIG applies wholesale (observability-config.ts). A malformed or unreadable file silently falls back to defaults. Merging is a deep merge where objects merge key-by-key but arrays and scalars replace wholesale — so setting disabled_lenses or rule_overrides overwrites the default rather than extending it.

The shipped example file is misleading The repo's .scaffold/observability.yaml contains only knowledge_freshness.link_check.skip and a comment claiming "other observability features read configuration from elsewhere." Divergences flagged during authoring

The audit's own rule — "code is ground truth where it disagrees with the spec or the docs" — surfaced these mismatches while this page was written. They are deferred findings for a future round to resolve in the code/docs, not on this page:

  • pr_opened vs pr_open. CLAUDE.md abbreviates the event type as pr_open; the code only emits pr_opened. (§3)
  • Three engine verdicts, not four. needs-user-decision is an MMR review verdict; the audit engine emits only pass / degraded-pass / blocked. (§6)
  • Fix-dispatcher security note conflated. CLAUDE.md's --fix note calls the dispatcher "not project-config-overridable"; in fact fix.dispatcher_command is configurable, while it's the Lens H LLM dispatcher that is hard-coded. (§10, §5)
  • pr_review_stale reserved. The sixth stall signal has a config default but is not yet emitted (deferred pending a per-PR correlation_id on the mmr adapter). (§7)

  • No F-scope.wave_budget key. CLAUDE.md implies a wave budget in observability.yaml; Lens F reads wave/phase budget from .scaffold/state.json instead. (§13)
  • ui_glob default narrower than spec. Spec default covers {tsx,jsx,vue,svelte} + styles; shipped default is just src/components/**/*.tsx. (§13)
