Executive overview

What this system does

Knowledge entries declare a volatility tier and a list of sources. A daily cron prefilters at most ten entries that are due (by cadence or by source-hash change), runs a grounded LLM audit, opens one PR per drifted entry, and gates the PR on five checks. In parallel, downstream agents emit knowledge_gap_signal events when they hit topics not covered by the KB; Lens I aggregates those signals into P1/P2 findings, suppressing topics that an entry already covers.

04

The pipeline

From candidate to merged PR

The cron is a thin bash loop — the brains live in three CLI subcommands and a meta-prompt that runs a grounded LLM against pre-fetched source bodies.

Prefilter

An entry becomes a candidate when (1) it has at least one source, AND (2) either its last-reviewed is older than the cadence window, OR a source's prefetched hash differs from the stored one. Priority orders newest-first. Top --max wins.

Hash check is a tiebreaker, not a baseline. Entries that are already past their cadence window (ageDays > window) are selected immediately at priority 50 + ageDays — no network cost paid. The hash check only runs when an entry is still inside its window (the else branch at audit-prefilter.ts:39-62), and runs Promise.all over the entry's 1–3 sources, swallowing fetch errors so a slow upstream doesn't crash the cron. Missing-last-reviewed entries skip both checks and select at priority 100 (highest).

src/knowledge-freshness/audit-prefilter.ts:14-72
for (const e of entries) {
  if (e.sources.length === 0) continue            // no sources = no audit
  if (!e.lastReviewed)        { select = true; priority = 100 }
  else if (ageDays > window) { select = true; priority = 50 + ageDays }
  else {
    // hash check — Promise.all over a small per-entry list (1-3 sources)
    if (anyHashChanged) { select = true; priority = 75 }
  }
}
candidates.sort((a, b) => b.priority - a.priority)
return candidates.slice(0, max).map(c => c.entry)

Audit verdicts

The meta-prompt at content/tools/knowledge-audit-entry.md instructs the LLM to read pre-fetched source bodies (no web tool available) and emit one of four verdicts. Every verdict opens a PR (cron audit.yml:113-155): the dry-run apply runs first so gates can inspect the proposed diff, then --open-pr creates the branch. The cards below describe what the PR contains for each verdict.

current

Sources confirm; no body findings. The PR is frontmatter-only: updates last-reviewed, sources[*].hash, sources[*].retrieved so the entry exits the prefilter queue. The verdict JSON is embedded in the PR body for provenance.

minor-drift

Wording slightly outdated; no substantive claim wrong. The applyVerdictToEntry contract refuses any proposed_changes on this verdict (audit-apply.ts:54-58) — the PR opens with frontmatter persistence and the findings table as commentary, no body edits. Reviewers can manually copy excerpts if they want to apply touchups.

major-drift

Substantive claim now inaccurate. Body edits land via proposed_changes (H2-heading-anchored splices). Gate 4 (anti-over-rewrite) blocks the PR if a stable entry's diff exceeds 20% line churn without the override:anti-over-rewrite label.

superseded

A new edition/version has shipped; version-pin needs to advance. last-reviewed does NOT advance for this verdict (audit-apply.ts:103-118) — only hash/retrieved update. The entry stays due in the next prefilter run until a human re-audits against the new edition and produces a follow-up current or major-drift verdict. This prevents a known-stale entry from looking fresh.

PR generation

Branch: knowledge-freshness/<entry>-<YYYY-MM-DD>. The body embeds the verdict JSON, the findings table, and a per-source confirmation row. Each candidate gets its own PR off origin/main — the cron explicitly git checkout main between iterations (audit.yml:151-165) and restores the entry with git checkout -- <path> between the dry-run apply (for gates) and the final --open-pr call. PRs do not stack; failures isolate per-candidate.

VERSION bump on merge (dedicated workflow)

A separate workflow at .github/workflows/knowledge-freshness-version-bump.yml fires on PR closed (merged-only) when the source branch starts with knowledge-freshness/ or the PR carries the knowledge-freshness label. It computes the next SemVer from the PR title and body, writes content/knowledge/VERSION, commits with the prefix chore(knowledge): (deliberately not knowledge-freshness/*) so the commit doesn't re-trigger itself, then git pull --rebase before pushing. Bump rules (src/knowledge-freshness/bump-version.ts:26-45):

MatchBumpNotes
BREAKING CHANGE: anywhere in title or body (start-of-line in body)majorWins over every other prefix.
feat(knowledge): or feat(knowledge-freshness): title prefixminorCase-sensitive.
chore(knowledge): or chore(knowledge-freshness): title prefixpatchUsed by the bump commit itself.
Anything else (including fix(knowledge):)patchWorkflow logs a ::notice:: for unrecognized prefixes.

The start-of-line anchor on the BREAKING CHANGE body match (/^BREAKING CHANGE:/m) is deliberate — a freshness PR's body embeds an LLM-generated findings table whose evidence excerpts could otherwise mention "BREAKING CHANGE:" and trigger an accidental major bump.

CI gates

The five gates a freshness PR must pass

The cron's GITHUB_TOKEN-opened PRs don't fire downstream workflows, so the cron also runs the gate code inline (same CLI surface). Human-opened freshness PRs get gated by the workflow.

The gate workflow's anti-tamper checkout

The workflow does not build the gate code from origin/main — it builds from HEAD. The intended hardening (build from base, overlay PR HEAD's content/knowledge/ only) is deferred because the Phase 2 bootstrap PR introduced the gate code itself. Risk is mitigated by mandatory PR review until a follow-up flips the checkout strategy.

Lens I — gap detection + suppression

How signals become findings (and how findings become silent)

Lens I runs under --scope=docs and --scope=all. It collects signals from the ledger (90-day window) plus synthetic signals from tasks/lessons.md, buckets them by normalized topic, applies the threshold matrix, and suppresses buckets whose topic is already covered by an entry.

Threshold matrix

Drag the sliders below the grid to highlight a cell. Cells lit yellow are P2; cells lit orange are P1.

Rules from src/observability/checks/lens-i-knowledge-gaps.ts:148-149: signalCount ≥ 5 && distinctProjects ≥ 3 → P1; else signalCount ≥ 3 && distinctProjects ≥ 2 → P2; else no finding.

Topic normalization

Lens I normalizes the topic before bucketing, then validates the result. Two distinct steps — normalizeTopic always produces a (possibly empty) string; isValidTopic decides whether to accept it. Try inputs that exercise each path:

normalizeTopic at src/observability/checks/lens-i-lessons-scanner.ts:32-38; isValidTopic at :114-116. The function lowercases, strips apostrophes, replaces every other non-slug run with a single hyphen, collapses repeats, and trims. The validator additionally enforces ≤ 80 chars and the regex /^[a-z0-9]+(-[a-z0-9]+)*$/.

    What the lessons.md scanner sees

    Lens I synthesizes signals from tasks/lessons.md at audit time (read inline, no ledger writes). Two passes per non-fenced line (code-fenced blocks are skipped):

    1. Explicit marker — <!-- gap-topic: <slug> --> (slug must already be kebab-case; the marker regex enforces it). Multiple markers per line allowed.
    2. Heuristic phrases (case-insensitive, sentence-terminating .!? end the capture; quotes/backticks terminate unconditionally):
      • "would have helped to have a guide on X"
      • "missing knowledge entry for X"
      • "no knowledge entry for X" / "no kb entry for X"
      • "missing knowledge: X"

    Captured topics run through the same normalizeTopic shown above, then through isValidTopic (≤80 chars, kebab-case). Synthetic signals carry project_id: "lessons" and are excluded from the distinct-projects count by the aggregator's delete('lessons') rule (decision #6) — they corroborate but don't independently satisfy the threshold.     Provider selection

    Anthropic vs DeepSeek (cron uses DeepSeek)

    The cron switched to DeepSeek HTTP in PR #393 to remove the local claude CLI dependency from CI. Local audits keep using whichever provider is configured. Precedence: explicit flag > env var > inferred from API key > claude on PATH.

    Anthropic

    Subprocess: claude -p --tools "" (the empty-tools flag disables WebFetch and every other built-in so the model can only read the prefetched source bodies, never re-fetch). Requires the claude CLI on PATH regardless of how the provider was chosen — the resolver throws (providers/index.ts:44-56) if anthropic is picked via flag, env, or API-key inference and claude isn't installed. ANTHROPIC_API_KEY alone is not sufficient. Auth via claude /login keychain OR ANTHROPIC_API_KEY — but the CLI install is non-negotiable.

    src/knowledge-freshness/providers/anthropic.ts

    DeepSeek

    HTTP. Requires DEEPSEEK_API_KEY. Default model: deepseek-v4-flash; override via KNOWLEDGE_FRESHNESS_DEEPSEEK_MODEL to deepseek-v4-pro (other values throw at dispatcher-build time per providers/deepseek.ts:54-58). Hardcoded thinking: { type: 'disabled' } — v4 ignores temperature: 0 when thinking is on, and chain-of-thought before structured JSON wastes the 8192-token budget. URL is hardcoded to https://api.deepseek.com/chat/completions; project-local config cannot redirect (decision #7 invariant). No subprocess; works in CI without installing the Claude CLI.

    src/knowledge-freshness/providers/deepseek.ts

    Precedence rules

    1. --provider <name> — explicit flag, operator override
    2. KNOWLEDGE_FRESHNESS_PROVIDER env var
    3. Single API key in env — inferred
    4. Both API keys present → error (ambiguous)
    5. No env, claude on PATH → anthropic (subprocess uses keychain)
    6. Nothing → error (no provider configured)

    From src/knowledge-freshness/providers/index.ts:36-109. The runtime check at line 44-56 catches anthropic-via-any-path without claude installed so the operator gets a precise install message instead of command-not-found at first audit.

    CLI reference

    Every command that touches the system

    All commands ship in the published CLI. Expand a row for synopsis, flags, exit codes, and an example.

    scaffold knowledge-freshness audit-prefilter [--max=N] pick at most N candidates due for audit

    Walks content/knowledge/, applies cadence + hash check, prints JSON array of candidates to stdout. The cron pipes this into jq -c '.[]'.

    Flags: --max=<n> (default 10, positive integer; otherwise the call returns an empty list to enforce the ceiling)

    Exit codes: 0 on success; non-zero on filesystem error.

    example
    node dist/index.js knowledge-freshness audit-prefilter --max=10 \
  | jq '.[] | {name, path, priority}'
    scaffold knowledge-freshness audit-run-entry <path> run grounded audit; emit verdict JSON

    Pre-fetches each declared source through SSRF/DNS/redirect/timeout guards, dispatches the grounded audit prompt, prints verdict JSON to stdout.

    Flags: --provider anthropic|deepseek (overrides the env precedence)

    Env: DEEPSEEK_API_KEY (deepseek), ANTHROPIC_API_KEY (optional for anthropic; claude on PATH still required), KNOWLEDGE_FRESHNESS_DEEPSEEK_MODEL (allowlisted overrides)

    example
    DEEPSEEK_API_KEY=sk-... node dist/index.js knowledge-freshness \
  audit-run-entry content/knowledge/core/security-best-practices.md \
  > /tmp/verdict.json
jq .verdict /tmp/verdict.json   # → "current" | "minor-drift" | …
    scaffold knowledge-freshness audit-apply <path> <verdict.json> [--open-pr] apply verdict to disk; optionally open PR

    Patches the entry's frontmatter (last-reviewed, sources[*].hash, sources[*].retrieved) and applies proposed changes by H2 heading. With --open-pr, creates a branch and opens a PR via gh.

    Hash provenance: the CLI wrapper re-fetches every verdict.sources_checked.url and computes its own sha256, then passes the map to applyVerdictToEntry via trustedHashes (audit-apply.ts:36-46). The persisted sources[*].hash is therefore deterministic, not the LLM's claim. Unit tests that bypass the wrapper fall back to the LLM-claimed value.

    Coverage check: apply refuses to advance last-reviewed if the verdict's sources_checked doesn't cover every declared source (matching on url + anchor when an anchor is set; loose match otherwise). Prevents partial audits from marking entries fresh.

    example
    node dist/index.js knowledge-freshness \
  audit-apply content/knowledge/core/<name>.md /tmp/verdict.json --open-pr
    scaffold observe audit --lens I-knowledge-gaps [--knowledge-root <path>] [--fix] run Lens I; suppress against KB; optionally dispatch fixes

    Runs the gap-detection lens against the local ledger + tasks/lessons.md. --knowledge-root overrides yaml + auto-detect for suppression. --fix dispatches the fix flow on blocking findings; the override threads through initial + per-finding verifier + postfix audits.

    example
    # local audit of a downstream project, with the override pointing at the dev KB
scaffold observe audit --lens I-knowledge-gaps \
  --knowledge-root /Users/me/scaffold/content/knowledge --scope=docs
    scaffold observe event knowledge_gap_signal --topic=<slug> --source=<…> --project-id=<sha> … emit a gap signal manually

    Writes one validated event to the ledger. Used by the assembly-time tail and by operators backfilling synthetic signals.

    Validation: topic is kebab-case ≤80 chars; source ∈ {agent_search, lessons, manual}; project_id is 64-char sha256 hex OR literal lessons (when source=lessons).

    example
    PROJECT_ID=$(git remote get-url origin | shasum -a 256 | awk '{print $1}')
scaffold observe event knowledge_gap_signal \
  --branch=main --topic=agent-eval-harnesses --source=agent_search \
  --project-id="$PROJECT_ID" --step-name=tech-stack \
  --agent-excerpt="No knowledge entry for agent eval harnesses"
    scaffold observe ack <prefix-or-id> acknowledge or reopen a finding

    Marks a finding acknowledged so it stops surfacing in the "Needs Attention" surface. Use when a Lens I finding represents a topic you deliberately don't want to cover.

    example
    scaffold observe ack a3f2 --status acknowledged --note "covered by docs/playbook"
    make validate-knowledge Gate 1 of 5 — frontmatter schema

    Runs the Zod schema validator over every entry under content/knowledge/. README files are excluded. Wraps node dist/index.js validate-knowledge so the cron and the gate workflow use the same CLI surface.

    Gate-side subcommands (also runnable locally for triage)

    scaffold knowledge-freshness link-check [<path>] [--files-from <json>] Gate 2 — every sources[*].url returns 2xx

    HTTP-HEADs every sources[*].url declared in the target entries. Treats 2xx as pass; everything else (including redirects that don't resolve to a 2xx terminal) as fail.

    Inputs: a single path argument or --files-from <json> (a JSON-array file of paths; the gate workflow uses this form to avoid bash-interpolating filenames that could contain shell metacharacters).

    Exit codes: 0 on all-pass, 1 on any failure.

    example
    node dist/index.js knowledge-freshness link-check \
  content/knowledge/core/security-best-practices.md
    scaffold knowledge-freshness lint-unsourced [<path>] [--files-from <json>] [--diff <patch>] Gate 3 (advisory) — flag new normative claims missing a source

    Heuristic scan for normative language ("must", "should", "never", "always") in new lines of the diff that don't reference a sources[] URL. Advisory: prints findings but always exits 0 in CI (per spec §A.5).

    Flags: --files-from <json> (gate workflow form), --diff <patch> (so the gate evaluates only added/changed lines, not the whole entry).

    scaffold knowledge-freshness anti-over-rewrite [--files-from <json>] [--diff <patch>] [--pr-labels <csv>] Gate 4 — stable entries can't lose >20% lines without override label

    For each changed entry whose frontmatter declares volatility: stable, compares the diff's deleted-line count against 20% of the original body. If the threshold is crossed AND --pr-labels does NOT contain override:anti-over-rewrite, exit 1.

    The cron's inline gates pass --pr-labels "" deliberately — the cron cannot self-apply override labels, so cron-generated stable rewrites that cross 20% always block. Maintainers who genuinely need such a rewrite run audit-apply --open-pr locally and apply the label there (gate workflow re-runs on labeled/unlabeled events).

    example
    git diff origin/main...HEAD -- 'content/knowledge/**/*.md' > /tmp/pr.diff
node dist/index.js knowledge-freshness anti-over-rewrite \
  --pr-labels "$(gh pr view --json labels --jq '.labels[].name' | paste -sd ,)" \
  --files-from /tmp/changed.json \
  --diff /tmp/pr.diff
    scaffold knowledge-freshness deep-guidance-check [<path>] [--files-from <json>] Gate 5 — literal "## Deep Guidance" heading must survive

    Asserts each changed entry still contains a ## Deep Guidance heading. The assembly engine pulls just that section for downstream prompts (extractDeepGuidance); losing it breaks every downstream that depends on the entry. Match is case-sensitive (the assembly regex is case-insensitive in practice but the gate is stricter to catch near-misses like "## Deep Guidance (Updated)").

    scaffold knowledge-freshness bump-version --title <str> --body <str> VERSION bump dry-run (used by the version-bump workflow)

    Pure-function CLI wrapper around deriveBumpKind + bumpSemver from src/knowledge-freshness/bump-version.ts. Reads the current SemVer from content/knowledge/VERSION, computes the next version per the table in §4, and prints two lines to stdout:

    bump: patch
next: 0.1.1

    The workflow at .github/workflows/knowledge-freshness-version-bump.yml:77-103 parses these lines to keep the derivation rules in one place (the TS module) instead of duplicating in bash.

    Operations cheat sheet

    Four playbooks for things that go wrong

    SYMPTOM

    An entry's audit failed in the cron

    The cron logs audit failed for <name> — moving on and continues to the next candidate. Possible causes: provider auth (DEEPSEEK_API_KEY rotated), source URL now 404s, source body too large (96 KiB cap), LLM timeout. The entry stays in the queue for tomorrow.

    Fix:

    manual reproduction
    # reproduce locally (or in actions/runner)
DEEPSEEK_API_KEY=sk-… node dist/index.js knowledge-freshness \
  audit-run-entry content/knowledge/<cat>/<name>.md
# read stderr to see if it's an URL issue or provider issue

    SYMPTOM

    Lens I keeps surfacing a topic the KB already covers

    Lens I emits the finding because suppression didn't match. Either the resolver returned root: null (look for the [Lens I] knowledge-root not located stderr line), or the entry's name: doesn't normalize to the same slug as the bucket topic. The match is exact and post-normalize.

    Fix:

    diagnose
    # 1. Force-resolve the KB:
scaffold observe audit --lens I-knowledge-gaps --json \
  --knowledge-root /path/to/content/knowledge \
  | jq '.findings[] | select(.lens_id=="I-knowledge-gaps")'
# 2. If still surfaced, check the entry's name: matches the bucket topic exactly
grep -A1 "^---" content/knowledge/<cat>/<slug>.md | grep "^name:"

    SYMPTOM

    Downstream auto-detect can't find the KB

    An operator on their own project sees [Lens I] knowledge-root not located. findScaffoldKnowledgeRoot walks parents from the CLI install's module location (not the project's cwd), looking for package.json with name: '@zigrivers/scaffold'. If the install is symlinked or repackaged, the walk may miss.

    Fix: Set the path in .scaffold/observability.yaml (soft-fail tier 2):

    .scaffold/observability.yaml
    lenses:
  I-knowledge-gaps:
    knowledge_root: /opt/homebrew/lib/node_modules/@zigrivers/scaffold/content/knowledge

    SYMPTOM

    Yaml knowledge_root stops working after an upgrade

    The yaml tier soft-fails on validation errors and records { source: 'yaml', outcome: 'invalid', reason } in the resolution's attempts trail. Lens I appends that reason to the warning string, so look for yaml lenses.I-knowledge-gaps.knowledge_root '…' was invalid: … in stderr.

    Fix: Validation requires (1) directory exists, (2) is a directory, (3) contains a VERSION marker file, (4) loadKnowledgeIndex runs without throwing. Empty index is OK. The most common failure after an upgrade is that the install path moved — find / -name VERSION -path '*content/knowledge*' to locate the new path, then update the yaml.

    SYMPTOM

    A source URL fetches fine in curl but the cron rejects it

    The SSRF guard at src/knowledge-freshness/source-url-validator.ts re-resolves the hostname at fetch time and rejects any IP in a non-globally-routable range (RFC1918, link-local, loopback, CGNAT, ULA, IPv4-mapped IPv6, multicast, reserved, etc. — see :73-93 for the full IPv4 set and :116-130 for IPv6). Common cause: an internal DNS view returning a private IP for an outwardly public-looking hostname.

    Diagnose:

    manual reproduction
    # Show every IP the guard would see (uses dns.lookup, same path as fetch)
node -e 'require("node:dns").promises.lookup("<host>", { all: true }).then(console.log)'

# Trigger the validator directly to see the exact rejection message
node -e 'import("./dist/knowledge-freshness/source-url-validator.js").then(m => console.log(m.validateSourceUrl("https://<host>/path")))'

    Fix: either move the source to a globally-routable host (a permanent vendor docs URL) or remove it from sources[] until the DNS view changes. Adding the host to the allowlist does not bypass the SSRF guard — allowlisting is a separate, weaker "is this source authoritative?" check that only warns.

    Tunable config (.scaffold/observability.yaml)

    Everything operator-tunable about the system, in one place:

    .scaffold/observability.yaml (excerpt)
    lenses:
  I-knowledge-gaps:
    knowledge_root: /path/to/content/knowledge   # tier-2 resolver override

disabled_lenses: [I-knowledge-gaps]                 # opt-out

phase_audit:
  enabled: true                                   # default
  timeout_s: 60
  detached: false                                # fire-and-forget when true

fix:
  dispatcher_command: "claude -p"               # default
  timeout_s: 300
  per_finding_max_attempts: 3

    Loaded by loadObservabilityConfig at src/observability/engine/checks/observability-config.ts. The yaml's lenses.I-knowledge-gaps.knowledge_root is the tier-2 resolver input shown in §6. Anything outside this list is hardcoded (decision #7 invariant) so that an untrusted project can't redirect dispatch commands or LLM URLs at attacker-controlled targets.
