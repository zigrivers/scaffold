## Section 1: Domain Overview

The Assembly Engine is the core runtime orchestrator of scaffold v2. It replaces the original v2's three-layer prompt resolution system, mixin injection mechanics, and build-time assembly with a single runtime process that composes an assembled prompt from meta-prompts, knowledge base entries, project context, user instructions, and methodology settings. The assembly engine is invoked every time a user runs `scaffold run <step>` — there is no separate build step.

**Role in the v2 architecture**: The assembly engine sits at the center of the system. It consumes outputs from nearly every other domain — frontmatter schemas (domain 08) to parse meta-prompts, dependency resolution (domain 02) to check prerequisites, pipeline state (domain 03) to read completion status and write results, config validation (domain 06) for project metadata, methodology resolution (domain 16) for depth and enablement, and pipeline locking (domain 13) to prevent concurrent execution. It produces the assembled prompt — a deterministic 7-section document (ADR-045) that the AI reads, interprets, and executes in a single turn.

**Central design challenge**: The assembly engine must be deterministic — given identical inputs (meta-prompt, knowledge base, project context, instructions, depth), it must produce an identical assembled prompt. This is distinct from the AI's interpretation of the assembled prompt, which is bounded non-deterministic (ADR-041). The engine must also handle update mode (re-running a step on existing artifacts), gracefully degrade when optional inputs are missing (user instructions, knowledge base files), and remain fast enough to meet the 500ms assembly time budget (PRD §18).

## Section 2: Glossary

**assembled prompt** — The complete 7-section document produced by the assembly engine. Contains system framing, meta-prompt content, knowledge base entries, project context, methodology settings, user instructions, and execution instruction. Consumed by the AI in a single turn.

**assembly input** — The bundle of data passed to the assembly engine: step name, inline instructions (if any), CLI flags, and the resolved project root path.

**context gathering** — Step 4 of the execution sequence. The process of collecting prior artifacts, config, state, and decisions into the project context section of the assembled prompt.

**depth** — An integer from 1 to 5 controlling output rigor for a step. Resolved by domain 16. Included in the methodology section of the assembled prompt along with the meta-prompt's scaling guidance for that depth level.

**dry-run mode** — A mode where the assembly engine constructs and outputs the assembled prompt without executing it or updating state. Enabled by `--dry-run` flag.

**execution sequence** — The 9-step process the assembly engine follows: load meta-prompt, check prerequisites, load knowledge base, gather context, load instructions, determine depth, construct assembled prompt, AI generates and executes, update state.

**inline instructions** — Ephemeral user instructions passed via the `--instructions "..."` flag on a single `scaffold run` invocation. Highest precedence in the instruction hierarchy.

**knowledge base entry** — A markdown document in the `knowledge/` directory containing domain expertise on a specific topic. Referenced by meta-prompt frontmatter's `knowledge-base` field. Loaded in step 3 of the execution sequence.

**meta-prompt** — A compact markdown file (30-80 lines) in the `pipeline/` directory declaring a step's purpose, inputs, outputs, quality criteria, and methodology scaling guidance. Does not contain executable prompt text — that is generated by the AI from the assembled prompt.

**mode detection** — The process of determining whether a step is running in create mode (no prior artifacts) or update mode (existing artifacts found). Affects the project context section.

**project context** — The collection of prior artifacts, config.yml, state.json, and decisions.jsonl included in section 4 of the assembled prompt. Provides the AI with the project's current state.

**section** — One of the 7 fixed-order parts of an assembled prompt, as defined in ADR-045: system, meta-prompt, knowledge base, context, methodology, instructions, execution.

**update mode** — When a step is re-run on existing artifacts. The assembly engine includes the existing artifact content in the project context section, and the meta-prompt's Mode Detection block instructs the AI to diff and propose targeted updates rather than regenerating from scratch.

**user instructions** — Optional customization provided by the user at three layers: global (`.scaffold/instructions/global.md`), per-step (`.scaffold/instructions/<step>.md`), and inline (`--instructions` flag). Later layers override earlier ones on conflict.

## Section 3: Entity Model

```typescript
/**
 * Value object: a step name within the pipeline.
 * Must match a filename (without extension) in the pipeline/ directory.
 * Invariant: matches /^[a-z0-9]+(-[a-z0-9]+)*$/ (kebab-case).
 */
type StepName = string;

/**
 * Value object: a relative file path from the project root.
 * Used for artifact paths, instruction file paths, knowledge base paths.
 * Invariant: no leading slash, no '..' traversal, forward slashes only.
 */
type RelativePath = string;

/**
 * Value object: depth level for a pipeline step.
 * Invariant: integer in range [1, 5].
 */
type DepthLevel = 1 | 2 | 3 | 4 | 5;

/**
 * Value object: one of the 7 sections in the assembled prompt.
 * Order is fixed and must not change (ADR-045).
 */
type SectionKind =
  | 'system'
  | 'meta-prompt'
  | 'knowledge-base'
  | 'context'
  | 'methodology'
  | 'instructions'
  | 'execution';

/**
 * The input to the assembly engine.
 * Constructed by the CLI from command-line arguments and project state.
 *
 * Invariants:
 * - stepName must correspond to a file in pipeline/<stepName>.md
 * - projectRoot must be an absolute path to a directory containing .scaffold/
 */
interface AssemblyInput {
  /** The step to assemble and execute */
  stepName: StepName;

  /** Absolute path to the project root (contains .scaffold/ directory) */
  projectRoot: string;

  /** Inline instructions from --instructions flag. Empty string if not provided. */
  inlineInstructions: string;

  /** If true, output assembled prompt without executing or updating state */
  dryRun: boolean;

  /** If true, override advisory lock contention */
  force: boolean;

  /** Actor identity for state attribution (e.g., "agent-1") */
  actor: string;
}

/**
 * A loaded and parsed meta-prompt file.
 * Loaded from pipeline/<stepName>.md in step 1 of the execution sequence.
 *
 * Invariants:
 * - frontmatter and body are both present (frontmatter may have optional fields)
 * - body contains at minimum: purpose, inputs/outputs, quality criteria, scaling guidance
 */
interface MetaPromptFile {
  /** Step name derived from filename (without .md extension) */
  stepName: StepName;

  /** Parsed YAML frontmatter */
  frontmatter: MetaPromptFrontmatter;

  /** Markdown body content (everything after the closing --- delimiter) */
  body: string;
}

/**
 * Parsed frontmatter from a meta-prompt file.
 * Defines the metadata contract between meta-prompt authors and the assembly engine.
 * Schema defined in domain 08; this interface mirrors what the assembly engine consumes.
 */
interface MetaPromptFrontmatter {
  /** Human-readable description for display */
  description: string;

  /** Steps that must complete before this step can run */
  dependencies: StepName[];

  /** Pipeline phase for display grouping */
  phase: number;

  /** Artifact paths this step produces */
  outputs: RelativePath[];

  /** Knowledge base entry names to load during assembly */
  knowledgeBase: string[];

  /** Whether this step is conditional ("if-needed") or always active (null) */
  conditional: 'if-needed' | null;

  /** Argument hint for help output */
  argumentHint?: string;
}

/**
 * A loaded knowledge base entry.
 * Loaded from knowledge/<topic>.md in step 3 of the execution sequence.
 *
 * Invariants:
 * - content is non-empty (a knowledge base file with no content is an error)
 * - name matches the filename without extension
 */
interface KnowledgeBaseEntry {
  /** Entry name derived from filename (e.g., "domain-modeling") */
  name: string;

  /** The full markdown content of the knowledge base file */
  content: string;

  /** Subdirectory within knowledge/ (e.g., "core", "review") */
  category: string;
}

/**
 * Project context gathered in step 4 of the execution sequence.
 * Contains all project state relevant to the current step.
 *
 * Invariants:
 * - config is always present (assembly cannot proceed without config.yml)
 * - state is always present (assembly cannot proceed without state.json)
 * - artifacts may be empty (first step in pipeline has no prior artifacts)
 * - decisions may be empty (no decisions recorded yet)
 */
interface ProjectContext {
  /** Validated config from .scaffold/config.yml */
  config: ProjectConfig;

  /** Current pipeline state from .scaffold/state.json */
  state: PipelineStateSnapshot;

  /** Prior artifacts from completed steps, keyed by file path */
  artifacts: ArtifactEntry[];

  /** Decision log entries from .scaffold/decisions.jsonl */
  decisions: DecisionEntry[];

  /** Existing output artifact for this step, if any (triggers update mode) */
  existingOutput: ExistingArtifact | null;
}

/**
 * A snapshot of the pipeline state relevant to assembly.
 * Extracted from the full PipelineState (domain 03) with only
 * the fields the assembly engine needs.
 */
interface PipelineStateSnapshot {
  /** Which steps are completed, skipped, pending, or in_progress */
  stepStatuses: Record<StepName, StepStatus>;

  /** The methodology name from state.json */
  methodology: string;

  /** Init mode (greenfield, brownfield, v1-migration) */
  initMode: string;
}

/**
 * Value object: status of a step in the pipeline state.
 */
type StepStatus = 'pending' | 'in_progress' | 'completed' | 'skipped';

/**
 * Value object: a project config snapshot for assembly.
 * Extracted from the validated config (domain 06).
 */
interface ProjectConfig {
  /** Config schema version */
  version: number;

  /** Methodology name (deep, mvp, custom) */
  methodology: string;

  /** Project metadata */
  project: { name: string; platforms: string[] };

  /** Target platform adapters */
  platforms: string[];
}

/**
 * A prior artifact included in the project context section.
 * The assembly engine loads completed steps' output artifacts.
 */
interface ArtifactEntry {
  /** Relative path from project root (e.g., "docs/prd.md") */
  path: RelativePath;

  /** The step that produced this artifact */
  producedBy: StepName;

  /** File content (full text) */
  content: string;
}

/**
 * An existing output artifact for the current step.
 * Present when re-running a completed step (triggers update mode).
 */
interface ExistingArtifact {
  /** Relative path from project root */
  path: RelativePath;

  /** File content of the existing artifact */
  content: string;

  /** Timestamp when the step was last completed */
  completedAt: string;

  /** Depth level used in the previous execution */
  previousDepth: DepthLevel;
}

/**
 * A decision log entry from decisions.jsonl.
 */
interface DecisionEntry {
  /** Sequential ID (e.g., "D-001") */
  id: string;

  /** Step that produced this decision */
  step: StepName;

  /** Decision text */
  text: string;

  /** ISO 8601 timestamp */
  timestamp: string;

  /** Whether the step completed (false = provisional from crashed session) */
  promptCompleted: boolean;
}

/**
 * Resolved user instructions from all three layers.
 * Loaded in step 5 of the execution sequence.
 *
 * Invariants:
 * - Each layer is independently optional (may be empty string)
 * - Assembly order is: global, then per-step, then inline (later overrides earlier)
 */
interface UserInstructions {
  /** Content from .scaffold/instructions/global.md (empty if file doesn't exist) */
  global: string;

  /** Content from .scaffold/instructions/<step>.md (empty if file doesn't exist) */
  perStep: string;

  /** Content from --instructions flag (empty if not provided) */
  inline: string;
}

/**
 * A single section of the assembled prompt.
 * The assembled prompt is an ordered array of these sections.
 */
interface PromptSection {
  /** Which section this is (determines position in assembled prompt) */
  kind: SectionKind;

  /** Section header text (e.g., "## System", "## Knowledge Base") */
  header: string;

  /** Section body content */
  content: string;
}

/**
 * The complete assembled prompt — the primary output of the assembly engine.
 * A deterministic concatenation of 7 sections in fixed order (ADR-045).
 *
 * Invariants:
 * - Exactly 7 sections, in the order defined by SectionKind
 * - Given identical inputs, produces identical output (deterministic assembly)
 * - System section is always first, execution section is always last
 * - Knowledge base section contains individually delimited entries
 * - User instructions appear after all other content except execution
 */
interface AssembledPrompt {
  /** The 7 sections in fixed order */
  sections: [
    PromptSection & { kind: 'system' },
    PromptSection & { kind: 'meta-prompt' },
    PromptSection & { kind: 'knowledge-base' },
    PromptSection & { kind: 'context' },
    PromptSection & { kind: 'methodology' },
    PromptSection & { kind: 'instructions' },
    PromptSection & { kind: 'execution' },
  ];

  /** Metadata about the assembly for debugging and logging */
  metadata: AssemblyMetadata;
}

/**
 * Metadata about the assembly process.
 * Included in the AssemblyResult for logging and debugging.
 * Not included in the assembled prompt text itself.
 */
interface AssemblyMetadata {
  /** Step that was assembled */
  stepName: StepName;

  /** Depth level used */
  depth: DepthLevel;

  /** Whether update mode was triggered */
  updateMode: boolean;

  /** Knowledge base entries loaded (by name) */
  knowledgeBaseEntries: string[];

  /** User instruction layers that were non-empty */
  instructionLayers: ('global' | 'perStep' | 'inline')[];

  /** Number of prior artifacts included in context */
  artifactCount: number;

  /** Number of decision log entries included */
  decisionCount: number;

  /** Assembly duration in milliseconds */
  assemblyDurationMs: number;

  /** ISO 8601 timestamp when assembly completed */
  assembledAt: string;
}

/**
 * The result of assembly engine execution.
 * Contains the assembled prompt and assembly metadata.
 *
 * Invariants:
 * - If success is true, assembledPrompt is present and errors is empty
 * - If success is false, errors is non-empty
 * - Warnings may be present regardless of success/failure
 */
interface AssemblyResult {
  /** Whether assembly completed successfully */
  success: boolean;

  /** The assembled prompt (present only on success) */
  assembledPrompt?: AssembledPrompt;

  /** Fatal errors that prevented assembly */
  errors: AssemblyError[];

  /** Non-fatal warnings about the assembly */
  warnings: AssemblyWarning[];
}

/**
 * Prerequisite check result from step 2 of the execution sequence.
 * Determines whether the step can proceed or must be blocked.
 */
interface PrerequisiteCheckResult {
  /** Whether all prerequisites are satisfied */
  canProceed: boolean;

  /** If step is already completed, whether to offer update mode */
  alreadyCompleted: boolean;

  /** Depth used in previous completion (for update mode context) */
  previousDepth?: DepthLevel;

  /** Steps that are not yet completed (blockers) */
  unmetDependencies: StepName[];

  /** Whether the pipeline lock is held by another process */
  lockContention: boolean;

  /** Lock holder info if contention detected */
  lockHolder?: { pid: number; step: StepName; started: string };

  /** Whether the step is disabled in methodology config */
  stepDisabled: boolean;
}

/**
 * Update mode context — gathered when re-running a completed step.
 * Provides the AI with the existing artifact for diff-based updates.
 */
interface UpdateModeContext {
  /** The existing artifact content */
  existingArtifact: ExistingArtifact;

  /** Depth used in previous execution */
  previousDepth: DepthLevel;

  /** Depth being used in current execution */
  currentDepth: DepthLevel;

  /** Whether depth increased (triggers more thorough update) */
  depthIncreased: boolean;
}
```

## Section 4: State Transitions

The assembly engine itself does not have a persistent state machine — it runs to completion on each invocation. However, it orchestrates two state transitions in domain 03's pipeline state machine:

### Assembly Lifecycle (Per Invocation)

```
  [Invoked]
      |
      v
  Load Meta-Prompt ──> [Error: file not found] ──> FAIL (exit 1)
      |
      v
  Check Prerequisites
      |── Step disabled ──> FAIL (exit 1, "step disabled by methodology")
      |── Lock contention ──> FAIL (exit 3, unless --force)
      |── Unmet dependencies ──> FAIL (exit 2, list blockers)
      |── Already completed ──> Prompt "Re-run in update mode?" (or auto-proceed if --auto)
      |
      v
  Acquire Lock (domain 13)
      |
      v
  Set state to in_progress (domain 03)
      |
      v
  Load Knowledge Base ──> [Missing file: WARN, continue without it]
      |
      v
  Gather Project Context
      |── Load config ──> [Missing: FAIL (exit 1)]
      |── Load state ──> [Missing: FAIL (exit 1)]
      |── Load prior artifacts ──> [Missing: WARN, continue]
      |── Load decisions ──> [Missing: continue with empty]
      |── Check for existing output ──> [Found: set update mode]
      |
      v
  Load User Instructions ──> [Missing files: continue with empty]
      |
      v
  Determine Depth (domain 16)
      |
      v
  Construct Assembled Prompt (deterministic)
      |
      v
  [Dry-run?] ──yes──> Output assembled prompt ──> Release lock ──> EXIT 0
      |
      no
      |
      v
  AI Generates and Executes
      |
      v
  Update State (domain 03)
      |── Mark step completed
      |── Record depth used
      |── Verify artifacts exist
      |── Recompute next_eligible
      |
      v
  Record Decisions (domain 11)
      |
      v
  Release Lock (domain 13)
      |
      v
  Show Next Steps
      |
      v
  [Done]
```

### State Transitions Orchestrated

The assembly engine triggers these domain 03 state transitions:

1. **pending → in_progress**: Set when the assembly engine acquires the lock and begins execution (step 2). Sets the `in_progress` record with actor, timestamp, and step name.

2. **in_progress → completed**: Set after the AI finishes execution and artifacts are verified (step 9). Records completion timestamp, artifact verification status, depth used, and actor.

3. **pending → skipped**: Not triggered by the assembly engine directly. Handled by `scaffold skip` (domain 09).

If a crash occurs between transition 1 and 2, the pipeline state machine's crash recovery logic (domain 03) detects the stale `in_progress` record on the next invocation.

## Section 5: Core Algorithms

### Algorithm 1: Full Execution Sequence

The master algorithm that orchestrates all 9 steps.

```
function assembleAndExecute(input: AssemblyInput): AssemblyResult

  // Step 1: Load meta-prompt
  metaPrompt = loadMetaPrompt(input.projectRoot, input.stepName)
  if metaPrompt is null:
    return error(STEP_NOT_FOUND, "No meta-prompt found at pipeline/${input.stepName}.md")

  // Step 2: Check prerequisites
  prereqs = checkPrerequisites(input, metaPrompt)
  if not prereqs.canProceed:
    if prereqs.stepDisabled:
      return error(STEP_DISABLED, "Step ${input.stepName} is disabled by methodology config")
    if prereqs.lockContention and not input.force:
      return error(LOCK_CONTENTION, "Step ${prereqs.lockHolder.step} is running (PID ${prereqs.lockHolder.pid})")
    if prereqs.unmetDependencies is not empty:
      return error(UNMET_DEPENDENCIES, "Blocked by: ${prereqs.unmetDependencies.join(', ')}")

  // Handle already-completed step (update mode)
  updateMode = prereqs.alreadyCompleted

  // Acquire lock (domain 13)
  acquireLock(input.stepName, input.actor)

  // Set state to in_progress (domain 03)
  setStepInProgress(input.stepName, input.actor)

  // Step 3: Load knowledge base entries
  kbEntries = loadKnowledgeBase(input.projectRoot, metaPrompt.frontmatter.knowledgeBase)

  // Step 4: Gather project context
  context = gatherProjectContext(input, metaPrompt, updateMode)

  // Step 5: Load user instructions
  instructions = loadUserInstructions(input)

  // Step 6: Determine depth
  depth = resolveDepth(input.stepName)  // delegates to domain 16

  // Step 7: Construct assembled prompt
  assembled = constructAssembledPrompt(metaPrompt, kbEntries, context, instructions, depth, updateMode)

  // Dry-run: output and exit
  if input.dryRun:
    releaseLock()
    clearInProgress(input.stepName)  // revert to previous state
    return success(assembled, dryRun=true)

  // Step 8: AI generates and executes
  // (This happens outside the assembly engine — the assembled prompt
  //  is handed to the AI runtime. The assembly engine resumes at step 9.)

  // Step 9: Update state
  markStepCompleted(input.stepName, depth, input.actor)
  recordDecisions(input.stepName, input.actor)  // domain 11
  verifyArtifacts(metaPrompt.frontmatter.outputs)
  recomputeNextEligible()

  // Release lock (domain 13)
  releaseLock()

  // Show next available steps
  return success(assembled)
```

Complexity: O(P + K + A) where P = number of prerequisites to check, K = number of knowledge base entries to load, A = number of prior artifacts to gather. Dominated by file I/O, not computation.

### Algorithm 2: Context Gathering

Determines which prior artifacts to include in the project context section.

```
function gatherProjectContext(input, metaPrompt, updateMode): ProjectContext

  // Always load config and state
  config = readAndValidateConfig(input.projectRoot + "/.scaffold/config.yml")
  state = readState(input.projectRoot + "/.scaffold/state.json")

  // Load prior artifacts from completed steps
  artifacts = []
  for each stepName in state.stepStatuses:
    if state.stepStatuses[stepName] == 'completed':
      stepMeta = loadMetaPrompt(input.projectRoot, stepName)
      for each outputPath in stepMeta.frontmatter.outputs:
        fullPath = input.projectRoot + "/" + outputPath
        if fileExists(fullPath):
          artifacts.push({
            path: outputPath,
            producedBy: stepName,
            content: readFile(fullPath)
          })

  // Load decision log
  decisions = readDecisionLog(input.projectRoot + "/.scaffold/decisions.jsonl")

  // Check for existing output (update mode detection)
  existingOutput = null
  if updateMode:
    for each outputPath in metaPrompt.frontmatter.outputs:
      fullPath = input.projectRoot + "/" + outputPath
      if fileExists(fullPath):
        previousEntry = state.stepStatuses[input.stepName]
        existingOutput = {
          path: outputPath,
          content: readFile(fullPath),
          completedAt: previousEntry.at,
          previousDepth: previousEntry.depth
        }
        break  // use first found artifact for update mode context

  return { config, state, artifacts, decisions, existingOutput }
```

Complexity: O(S * O) where S = number of completed steps, O = average number of output artifacts per step. Bounded by the pipeline size (32 steps maximum).

### Algorithm 3: Assembled Prompt Construction

Builds the 7-section assembled prompt from gathered components.

```
function constructAssembledPrompt(
  metaPrompt, kbEntries, context, instructions, depth, updateMode
): AssembledPrompt

  sections = []

  // Section 1: System framing
  sections.push({
    kind: 'system',
    header: '## System',
    content: buildSystemSection(metaPrompt.stepName)
  })

  // Section 2: Meta-prompt (purpose, inputs, outputs, quality criteria)
  sections.push({
    kind: 'meta-prompt',
    header: '## Meta-Prompt: ${metaPrompt.stepName}',
    content: metaPrompt.body
  })

  // Section 3: Knowledge base entries (individually delimited)
  kbContent = ""
  for each entry in kbEntries:
    kbContent += "### Knowledge: ${entry.name}\n\n"
    kbContent += entry.content
    kbContent += "\n\n---\n\n"
  sections.push({
    kind: 'knowledge-base',
    header: '## Knowledge Base',
    content: kbContent
  })

  // Section 4: Project context
  contextContent = buildContextSection(context, updateMode)
  sections.push({
    kind: 'context',
    header: '## Project Context',
    content: contextContent
  })

  // Section 5: Methodology (depth + scaling guidance)
  methodologyContent = buildMethodologySection(depth, metaPrompt)
  sections.push({
    kind: 'methodology',
    header: '## Methodology',
    content: methodologyContent
  })

  // Section 6: User instructions (global + per-step + inline, in precedence order)
  instructionContent = buildInstructionsSection(instructions)
  sections.push({
    kind: 'instructions',
    header: '## User Instructions',
    content: instructionContent
  })

  // Section 7: Execution instruction (always last)
  sections.push({
    kind: 'execution',
    header: '## Execution',
    content: buildExecutionSection(updateMode)
  })

  return { sections, metadata: buildMetadata(...) }
```

### Algorithm 4: Update Mode Detection

Determines whether a step should run in create mode or update mode.

```
function detectUpdateMode(stepName, state, projectRoot, metaPrompt): UpdateModeContext | null

  entry = state.stepStatuses[stepName]

  // Step not previously completed — create mode
  if entry.status != 'completed':
    return null

  // Step completed — check if artifacts still exist
  anyArtifactExists = false
  for each outputPath in metaPrompt.frontmatter.outputs:
    if fileExists(projectRoot + "/" + outputPath):
      anyArtifactExists = true
      break

  if not anyArtifactExists:
    // Artifacts missing despite completed state — treat as create mode, warn
    emit warning(ARTIFACTS_MISSING, "Step ${stepName} is marked completed but artifacts are missing")
    return null

  // Update mode — existing artifacts found
  currentDepth = resolveDepth(stepName)  // domain 16
  return {
    existingArtifact: loadExistingArtifact(outputPath),
    previousDepth: entry.depth,
    currentDepth: currentDepth,
    depthIncreased: currentDepth > entry.depth
  }
```

### Algorithm 5: User Instruction Resolution

Loads user instructions from all three layers with precedence semantics.

```
function loadUserInstructions(input: AssemblyInput): UserInstructions

  instructionsDir = input.projectRoot + "/.scaffold/instructions/"

  // Layer 1: Global instructions (lowest precedence)
  globalPath = instructionsDir + "global.md"
  global = fileExists(globalPath) ? readFile(globalPath) : ""

  // Layer 2: Per-step instructions
  perStepPath = instructionsDir + input.stepName + ".md"
  perStep = fileExists(perStepPath) ? readFile(perStepPath) : ""

  // Layer 3: Inline instructions (highest precedence)
  inline = input.inlineInstructions

  return { global, perStep, inline }
```

Precedence is structural, not merge-based: all three layers are included in the assembled prompt in order (global first, inline last). The AI naturally gives more weight to later content, and the instructions section header documents the precedence rule explicitly.

### Algorithm 6: Knowledge Base Resolution

Resolves knowledge base references from meta-prompt frontmatter to loaded content.

```
function loadKnowledgeBase(projectRoot, references: string[]): KnowledgeBaseEntry[]

  entries = []
  knowledgeDir = projectRoot + "/knowledge/"  // or installed package path

  for each ref in references:
    // References are in format "category/name" (e.g., "core/domain-modeling")
    path = knowledgeDir + ref + ".md"

    if not fileExists(path):
      emit warning(KB_ENTRY_MISSING, "Knowledge base entry '${ref}' not found at ${path}")
      continue

    content = readFile(path)
    parts = ref.split("/")
    entries.push({
      name: parts[parts.length - 1],
      content: content,
      category: parts[0]
    })

  return entries
```

Missing knowledge base entries produce warnings, not errors. The assembly continues with whatever entries were found. This allows meta-prompts to reference optional knowledge entries without breaking the pipeline.

## Section 6: Error Taxonomy

```typescript
/**
 * Error codes for assembly engine failures.
 * Codes map to CLI exit codes per PRD §8.
 */
type AssemblyErrorCode =
  | 'STEP_NOT_FOUND'         // exit 1: no meta-prompt file for the requested step
  | 'STEP_DISABLED'          // exit 1: step disabled by methodology config
  | 'INVALID_FRONTMATTER'    // exit 1: meta-prompt frontmatter fails validation
  | 'CONFIG_MISSING'         // exit 1: .scaffold/config.yml not found
  | 'CONFIG_INVALID'         // exit 1: config.yml fails schema validation
  | 'STATE_MISSING'          // exit 1: .scaffold/state.json not found (run scaffold init)
  | 'UNMET_DEPENDENCIES'     // exit 2: prerequisite steps not completed
  | 'ARTIFACTS_NOT_FOUND'    // exit 2: required predecessor artifacts missing from disk
  | 'STATE_CORRUPTED'        // exit 3: state.json has invalid structure
  | 'LOCK_CONTENTION'        // exit 3: another step is running and --force not set
  | 'ASSEMBLY_FAILED'        // exit 5: assembled prompt construction failed
  ;

/**
 * A fatal error from the assembly engine.
 * Assembly cannot continue past this error.
 */
interface AssemblyError {
  /** Error code for programmatic handling */
  code: AssemblyErrorCode;

  /** Human-readable error message */
  message: string;

  /** Suggested recovery action */
  recovery: string;

  /** The step being assembled (if known at error time) */
  stepName?: StepName;

  /** CLI exit code */
  exitCode: number;
}

/**
 * Warning codes for non-fatal assembly issues.
 * Assembly continues despite these warnings.
 */
type AssemblyWarningCode =
  | 'KB_ENTRY_MISSING'       // knowledge base entry referenced but not found
  | 'INSTRUCTION_EMPTY'      // instruction file exists but is empty
  | 'ARTIFACTS_MISSING'      // step marked completed but artifacts not on disk
  | 'PROVISIONAL_DECISIONS'  // decision log entries with promptCompleted=false
  | 'DEPTH_CHANGED'          // re-running at different depth than previous execution
  | 'LARGE_CONTEXT'          // assembled prompt exceeds recommended size
  ;

/**
 * A non-fatal warning from the assembly engine.
 */
interface AssemblyWarning {
  /** Warning code for programmatic handling */
  code: AssemblyWarningCode;

  /** Human-readable warning message */
  message: string;

  /** The step being assembled */
  stepName: StepName;
}
```

## Section 7: Integration Points

### Domain 02 (Dependency Resolution) → This Domain

Domain 02 provides the dependency graph. The assembly engine uses `predecessors` to check whether all prerequisite steps are completed before allowing assembly to proceed (step 2 prerequisite check). The assembly engine does **not** re-run dependency resolution — it reads the precomputed graph.

### Domain 03 (Pipeline State Machine) → This Domain

Domain 03 provides the current pipeline state. The assembly engine reads `state.json` to:
- Determine step status (pending, completed, skipped, in_progress)
- Detect crash recovery scenarios (stale `in_progress` record)
- Identify which prior artifacts exist (for context gathering)
- Check completion status for update mode detection

### This Domain → Domain 03 (Pipeline State Machine)

The assembly engine writes state transitions to domain 03:
- Sets `in_progress` when beginning execution
- Sets `completed` with depth, timestamp, actor, and artifact verification after AI finishes
- Triggers `next_eligible` recomputation after state mutation

### Domain 06 (Config Validation) → This Domain

Domain 06 provides the validated configuration. The assembly engine reads `config.yml` for:
- Methodology name (used in methodology section)
- Project metadata (name, platforms — included in context section)
- Platform list (informational)

### Domain 08 (Frontmatter Schema) → This Domain

Domain 08 defines the meta-prompt frontmatter schema. The assembly engine parses frontmatter according to this schema to extract dependencies, outputs, knowledge-base references, conditional status, and phase.

### This Domain → Domain 11 (Decision Log)

After AI execution completes, the assembly engine records any decisions made during the step to `decisions.jsonl` via domain 11's write interface.

### Domain 13 (Pipeline Locking) → This Domain

Domain 13 provides the advisory lock mechanism. The assembly engine:
- Acquires the lock before setting `in_progress`
- Holds the lock during AI execution
- Releases the lock after state update completes
- Checks for stale locks on entry (dead PID detection)

### Domain 16 (Methodology & Depth Resolution) → This Domain

Domain 16 resolves the effective depth level for the current step. The assembly engine queries domain 16 for:
- Step enablement status (is the step active in the current methodology?)
- Effective depth (preset default or custom override)
- Scaling guidance extraction from the meta-prompt body

## Section 8: Edge Cases & Failure Modes

### 1. Re-running a completed step (update mode)

**Scenario**: User runs `scaffold run create-prd` when `create-prd` is already completed.
**Expected behavior**: Assembly engine detects existing artifact, enters update mode. Existing artifact content is included in the project context section. The meta-prompt's Mode Detection section instructs the AI to diff and propose targeted updates. State is updated with new completion timestamp and depth.

### 2. Running a step whose dependencies are skipped

**Scenario**: User skipped `review-prd` and runs `phase-01-domain-modeling` which depends on it.
**Expected behavior**: Assembly proceeds. Skipped steps are treated as "done" for dependency resolution (ADR-020). The skipped step's artifacts will not be in the project context (they don't exist). The AI generates output without that context.

### 3. Missing knowledge base file referenced in meta-prompt frontmatter

**Scenario**: A meta-prompt's `knowledge-base` field references `"review/review-domain-modeling"` but the file doesn't exist.
**Expected behavior**: Assembly emits a `KB_ENTRY_MISSING` warning and continues without that entry. The knowledge base section of the assembled prompt contains only the entries that were found. Assembly does not fail.

### 4. Missing user instruction files

**Scenario**: `.scaffold/instructions/global.md` does not exist.
**Expected behavior**: The global instruction layer is empty string. Assembly continues. The instructions section of the assembled prompt contains only the layers that have content. If all three layers are empty, the instructions section header is still present but with no content.

### 5. Assembly when project has no prior artifacts (first step)

**Scenario**: User runs `scaffold run create-prd` as the first step in a fresh pipeline.
**Expected behavior**: The project context section contains config, state (all steps pending), and empty decisions. The artifacts list is empty. No update mode. The AI generates the artifact from scratch.

### 6. Conditional step force-enabled by user

**Scenario**: `phase-04-database-schema` is conditional (`if-needed`) and was auto-disabled by init wizard, but user manually enables it in config.yml and runs `scaffold run phase-04-database-schema`.
**Expected behavior**: Assembly proceeds normally. The assembly engine checks enablement via domain 16, which respects user overrides. The step assembles and executes as if it were always enabled.

### 7. Crash during state update after successful AI execution

**Scenario**: AI finishes executing the assembled prompt and produces artifacts, but the CLI crashes before writing to `state.json`.
**Expected behavior**: The lock file remains (stale PID). The step's artifacts exist on disk but `state.json` still shows `in_progress`. On next invocation, domain 03's crash recovery detects the stale `in_progress` record, checks artifacts, finds them on disk, and marks the step completed. The stale lock is cleared.

### 8. Step with no knowledge base references

**Scenario**: A meta-prompt's `knowledge-base` field is empty or omitted.
**Expected behavior**: The knowledge base section of the assembled prompt has the header but no entries. Assembly continues. This is valid — some steps may not need external domain expertise.

### 9. Extremely large project context

**Scenario**: A late-pipeline step (e.g., `implementation-playbook`) has 20+ prior artifacts totaling hundreds of KB.
**Expected behavior**: Assembly emits a `LARGE_CONTEXT` warning if the assembled prompt exceeds a configurable threshold. All artifacts are still included. The warning is informational — it alerts the user that the context may approach AI token limits.

### 10. Concurrent assembly attempts

**Scenario**: Two agents try to run different steps simultaneously.
**Expected behavior**: The first agent acquires the lock. The second agent sees lock contention and fails with `LOCK_CONTENTION` (exit code 3) unless `--force` is set. Sequential execution is enforced (ADR-021).

## Section 9: Testing Considerations

### Key Test Categories

1. **Meta-prompt loading**: Test with valid frontmatter, missing required fields, malformed YAML, empty body, extra unknown fields (preserved per ADR-033).

2. **Prerequisite checking**: Test all prerequisite states: all met, some unmet, step disabled, lock contention, already completed, crash recovery scenario.

3. **Knowledge base loading**: Test with all entries found, some missing, all missing, empty references list, duplicate references.

4. **Context gathering**: Test with no prior artifacts, some artifacts, all artifacts, missing artifact files despite completed state, corrupt decisions.jsonl.

5. **User instruction loading**: Test with all three layers present, some missing, all missing, empty files, very large files.

6. **Assembled prompt construction**: Test section ordering is correct, delimiters are present, content is properly attributed, update mode context is included when applicable.

7. **Update mode detection**: Test with previously completed step (artifacts exist), completed step with missing artifacts, step at same depth, step at higher depth, step at lower depth.

8. **Determinism**: Run assembly twice with identical inputs and assert byte-identical output.

### Boundary Conditions

- First step in pipeline (no prior context)
- Last step in pipeline (maximum context)
- Step with zero dependencies
- Step with maximum dependencies (all prior steps)
- Knowledge base entry with very large content
- Inline instructions with special characters (quotes, newlines)
- Step name that contains only digits or hyphens

### What to Mock vs. Real Data

- **Mock**: File system reads (provide in-memory fixtures), domain 16 depth resolution (return fixed depth), lock acquisition (always succeed in unit tests).
- **Real data**: Integration tests should use real meta-prompt files and knowledge base entries from the `pipeline/` and `knowledge/` directories to catch format drift.

## Section 10: Open Questions & Recommendations

### Must Resolve Before Implementation

1. **Context window management**: How should the assembly engine handle context that exceeds the AI's token limit? The PRD specifies including all prior artifacts, but late-pipeline steps could produce assembled prompts exceeding 200K tokens. Options: (a) include everything and trust the AI to handle it, (b) implement a token budget and truncate/summarize older artifacts, (c) use the meta-prompt's `reads` field (if added) to specify exactly which prior artifacts are needed. **Recommendation**: Start with (a) — include everything — and add selective context loading later if token limits become a practical problem.

2. **Decision recording interface**: How does the assembly engine know which decisions the AI made during execution? The AI produces text output; decisions must be extracted and appended to `decisions.jsonl`. Options: (a) AI writes to decisions.jsonl directly via tool use, (b) assembly engine parses AI output for decision markers, (c) post-execution prompt asks AI to summarize decisions. **Recommendation**: (a) — the AI writes decisions via a scaffold-provided tool, matching v1 behavior.

3. **Artifact verification timing**: Should artifact verification happen immediately after AI execution (blocking) or lazily on next `scaffold run`? The PRD implies immediate verification (step 9). **Recommendation**: Immediate — verify all `outputs` paths exist on disk before marking completed.

### Can Defer

4. **Assembled prompt caching**: Could the assembly engine cache assembled prompts to avoid recomputation? Probably not worth it — assembly is fast (<500ms) and inputs change frequently.

5. **Parallel step execution**: ADR-021 mandates sequential execution. If parallel execution is ever desired, the assembly engine's lock acquisition logic would need to change to per-step locks rather than a global lock. Defer until user demand exists.

6. **Streaming assembly**: For very large assembled prompts, streaming the sections to the AI as they are constructed could reduce latency. Not needed until context window issues arise.
