## Section 1: Dashboard Overview

The Scaffold dashboard is a self-contained HTML file that provides the only GUI element in the Scaffold CLI. It visualizes pipeline progress, prompt status, and decision history for a Scaffold-managed project.

### Core Properties

| Property | Value |
|----------|-------|
| **Format** | Single HTML file with all CSS and JS inline |
| **External dependencies** | None — no CDN links, no `fetch()` calls, no iframes |
| **Generated by** | `scaffold dashboard` command |
| **Default output path** | `.scaffold/dashboard.html` |
| **Configurable output** | `--output <path>` flag |
| **Data sources** | `.scaffold/state.json`, `.scaffold/config.yml`, `.scaffold/decisions.jsonl` |
| **Server required** | No — opens as a local `file://` URL in the system browser |
| **Browser compatibility** | Modern evergreen browsers (Chrome, Firefox, Safari, Edge); no IE support |
| **Git status** | `.scaffold/dashboard.html` SHOULD be gitignored (generated artifact) |

### Lifecycle

1. User runs `scaffold dashboard`.
2. The CLI reads `state.json`, parses `config.yml`, and reads `decisions.jsonl`.
3. Dashboard Generator embeds all data as JSON within `<script>` tags in the HTML template.
4. The HTML file is written to `.scaffold/dashboard.html` (or the `--output` path).
5. Unless `--no-open` is passed, the CLI opens the file in the system default browser.
6. The dashboard is a snapshot — it reflects the state at generation time, not a live view.

### Theme Modes

The dashboard supports light and dark modes.

**System preference detection**: On first load, the dashboard reads the browser's `prefers-color-scheme` media query and initializes to the matching mode.

**Manual toggle**: A toggle button in the header bar switches between light and dark mode. The selected preference is stored in the browser's `localStorage` under the key `scaffold-dashboard-theme`. On subsequent opens, `localStorage` overrides the system preference.

**Non-localStorage path**: If `localStorage` is unavailable (e.g., strict privacy settings, `file://` restrictions in some browsers), the toggle still functions within the current session using an in-memory variable. The preference resets to system default on next open. The dashboard does not display an error or warning when `localStorage` is unavailable — it silently falls back to in-memory state.

## Section 2: Information Architecture

The dashboard is a single-page layout divided into six content sections, rendered top-to-bottom in the order described below.

### 2a: Header Bar

A fixed-height bar at the top of the page containing project-level metadata and controls.

| Element | Source | Position | Notes |
|---------|--------|----------|-------|
| Project name | `package.json` `name` field, or directory name if no `package.json` | Left | Plain text, monospace font |
| Methodology name | `config.yml` `methodology` field | Left, after project name | Displayed as a badge (e.g., `classic`) |
| Scaffold Version | `state.json` `scaffold-version` | Left, after methodology | Displayed as secondary text |
| Pipeline progress summary | Computed from `state.json` `prompts` map | Center | Text format: `{completed} of {total} complete — {percentage}%`. Example: `8 of 18 complete — 44%` |
| Light/dark mode toggle | User preference | Right | Icon button (sun/moon). See Section 1 for persistence behavior |

### 2b: Progress Overview

A summary section immediately below the header showing aggregate pipeline status.

**Progress bar**: A horizontal bar spanning the full content width. Segmented by status:
- Green segment: completed prompts (percentage of total)
- Gray segment: skipped prompts (percentage of total)
- Remaining: pending and in-progress prompts (unfilled)

**Summary cards**: Four cards in a horizontal row, each displaying a count and label:

| Card | Value | Color | Icon |
|------|-------|-------|------|
| Completed | Count of prompts with `status: "completed"` | Green | Checkmark |
| Skipped | Count of prompts with `status: "skipped"` | Gray | Arrow right |
| Pending | Count of prompts with `status: "pending"` | Neutral (muted text) | Circle outline |
| In Progress | Count of prompts with `status: "in_progress"` | Blue | Filled circle |

### 2c: Pipeline Phases

The primary content section. Displays prompts grouped by pipeline phase as a series of collapsible accordions.

**Phase accordion structure**:

```
┌─────────────────────────────────────────────────────┐
│ ▸ Phase 2 — Standards                   3/4 complete │
├─────────────────────────────────────────────────────┤
│  ✓  coding-standards    Establish coding conventions │
│  ✓  tdd-standards       Define TDD workflow          │
│  ✓  project-structure   Design directory layout      │
│  ○  git-workflow        Configure branching strategy  │
└─────────────────────────────────────────────────────┘
```

Each phase accordion header shows:
- Expand/collapse chevron (triangle icon)
- Phase number and name (from `phases[].index` and `phases[].name` in the dashboard data)
- Phase completion ratio (e.g., `3/4 complete`)

> **Data source note**: Phase data (`phases[].index`, `phases[].name`) is assembled at dashboard generation time from the methodology `manifest.yml`. Prompt descriptions are loaded from prompt frontmatter at generation time. Neither field exists in `state.json`.

**Prompt rows within a phase**: Each prompt is rendered as a row containing:

| Element | Source | Description |
|---------|--------|-------------|
| Status badge | `prompts.{slug}.status` | Icon + text badge. See badge design in Section 3b. |
| Prompt slug | Prompt key | Displayed in monospace font |
| Description | `prompts.{slug}` description or prompt frontmatter | Short description text |
| Timestamp | `prompts.{slug}.at` | Shown for completed, skipped, and in-progress prompts. Omitted for pending. Relative format preferred (e.g., "2 hours ago") with ISO timestamp on hover tooltip. |
| Skip reason | `prompts.{slug}.reason` | Shown inline (italic, muted) for skipped prompts only |

**Prompt detail expansion**: Each prompt row is expandable (click to toggle). The detail panel shows:

| Detail Field | Source | Description |
|--------------|--------|-------------|
| Description | Prompt metadata | Full description of what the prompt does |
| Produces | `prompts.{slug}.produces[]` | List of output file paths |
| Dependencies | Dependency graph from manifest | List of prerequisite prompt slugs |
| Completed by | `prompts.{slug}.completed_by` | Actor identity (for completed/skipped prompts) |
| Artifacts verified | `prompts.{slug}.artifacts_verified` | Boolean — whether output files have been confirmed on disk |

### 2d: What's Next Banner

A highlighted banner below the pipeline phases indicating the next action the user should take.

**Content when prompts are eligible**:
- Heading: "What's Next"
- Lists the next eligible prompt(s) from `state.json` `next_eligible[]`
- For each eligible prompt: slug, description, and prerequisite status (all prerequisites shown as green checkmarks)
- Command to run: `scaffold run` (displayed in a code block with a copy button)

**Content when all prompts are complete**:
- Heading: "Pipeline Complete"
- Message: "All prompts have been completed or skipped."

**Content when prompts are blocked**:
- Heading: "Blocked"
- Lists the blocking prompts and their current status
- No `scaffold run` command shown

### 2e: Decision Log Section

A chronological display of decisions recorded in `.scaffold/decisions.jsonl`.

**Grouping**: Decisions are grouped by prompt slug. Each group has a collapsible header showing the prompt slug and the count of decisions within.

**Decision entry display**:

| Field | Source | Description |
|-------|--------|-------------|
| Decision ID | `id` | Displayed as badge (e.g., `D-001`) |
| Decision text | `decision` | Full text of the decision |
| Timestamp | `at` | Relative format with ISO tooltip (same as prompt timestamps) |
| Actor | `completed_by` | Who made the decision |
| Category | `category` | Optional badge (e.g., `technology`, `architecture`). Omitted if absent. |
| Status indicator | `prompt_completed` | Confirmed decisions: normal styling. Provisional decisions (`prompt_completed: false`): styled with a warning indicator (amber left border and "Provisional" badge) |
| Review status | `review_status` | For decisions tagged `NEEDS_USER_REVIEW`: shows review status badge (`pending`, `approved`, `rejected`, `revised`). Pending review items display an amber "Needs Review" indicator. |

**Provisional decision styling**: Provisional decisions (`prompt_completed: false`) are rendered with:
- An amber left border (4px solid)
- A "Provisional" badge next to the decision ID
- Muted text opacity (0.8)
- Tooltip: "This decision was recorded during prompt execution but the prompt did not complete. The session may have crashed."

### 2f: Orphaned Entries Section (Conditional)

This section is rendered only when the state contains prompt entries whose slugs do not appear in the current resolved pipeline — typically after a methodology change.

**Visibility condition**: Shown when one or more keys in `state.json` `prompts` map do not match any prompt in the resolved pipeline (as computed at dashboard generation time).

**Content**:
- Heading: "Orphaned Entries"
- Explanatory text: "These prompts were completed or skipped under a previous methodology configuration but are no longer part of the current pipeline."
- Table of orphaned prompts:

| Column | Description |
|--------|-------------|
| Slug | The prompt slug (monospace) |
| Status | Last known status (completed/skipped) with badge |
| Completed at | Timestamp from `prompts.{slug}.at` |
| Actor | `prompts.{slug}.completed_by` |

**Styling**: The section uses a muted background and a dashed top border to visually separate it from the active pipeline content. No interactive elements — rows are not expandable.

## Section 3: Visual Design

### 3a: Design Language

The dashboard uses a clean, technical aesthetic appropriate for a developer tool. No playful icons, no rounded-corner card flourishes, no gradients.

**Typography**:

| Use | Font Stack | Weight | Size |
|-----|-----------|--------|------|
| Headings (h1, h2, h3) | System sans-serif (`-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto, sans-serif`) | 600 (semibold) | h1: 24px, h2: 20px, h3: 16px |
| Body text | System sans-serif (same stack) | 400 (regular) | 14px |
| Code / slugs / commands | System monospace (`"SF Mono", "Cascadia Code", "Fira Code", Consolas, monospace`) | 400 | 13px |
| Badges | System sans-serif | 500 (medium) | 12px |
| Small text (timestamps, hints) | System sans-serif | 400 | 12px |

**Layout**:

| Property | Value |
|----------|-------|
| Max content width | 1200px, centered |
| Horizontal padding | 24px on each side |
| Section spacing | 32px between major sections |
| Card gap | 16px |
| Row height (prompt rows) | 40px minimum, expandable |
| Border radius | 4px for cards and badges, 2px for inline elements |

**Spacing scale**: Based on a 4px base unit: 4, 8, 12, 16, 24, 32, 48, 64.

**Borders**: Minimal. Use 1px solid borders only for table rows and card containers. Prefer background color shifts and spacing for visual grouping.

### 3b: Status Badge Design

### 3d: Responsive Design

The dashboard is designed for desktop use but adapts to smaller viewports.

| Viewport | Layout Behavior |
|----------|----------------|
| >= 1200px | Full layout. Summary cards in a 4-column row. Progress bar full width. Max content width 1200px centered. |
| 768px – 1199px | Summary cards wrap to a 2x2 grid. Content fills viewport width with 24px padding. |
| < 768px | Summary cards stack vertically. Phase accordion headers truncate methodology name. Decision log groups collapse by default. Horizontal padding reduces to 16px. |

**Minimum supported viewport**: 768px. The dashboard displays a usable layout down to 768px width. Below 768px, layout may overflow horizontally but remains functional.

**Maximum supported viewport**: 1920px. Content is capped at 1200px width and centered; no content stretches beyond this.

### 3e: Progress Bar Design

The progress bar is a horizontal bar with a height of 8px and rounded ends (`border-radius: 4px`).

**Segments** (left to right):
1. **Completed** (green): width = `(completed / total) * 100%`
2. **Skipped** (gray): width = `(skipped / total) * 100%`
3. **Remaining** (unfilled / background): width = remainder

The bar uses CSS custom properties for colors: `var(--status-green-bg)` for completed, `var(--status-gray-bg)` for skipped, `var(--bg-tertiary)` for unfilled.

A percentage label is displayed above the right end of the bar: `{percentage}%`.

## Section 4: Interaction Patterns

The dashboard is read-only. No interaction writes data or modifies scaffold state.

### Accessibility (ARIA)

Interactive elements include the following ARIA attributes:

- **Phase accordion headers**: `role="button"` and `aria-expanded="true|false"` (reflects current expand/collapse state).
- **Status badges**: `aria-label` describing the status (e.g., `aria-label="Status: Done"`, `aria-label="Status: In Progress"`).
- **Skip-to-content link**: A visually hidden anchor (`<a href="#main-content" class="skip-link">Skip to content</a>`) is the first focusable element in the DOM. It becomes visible on focus and jumps past the header bar to the progress overview section (`id="main-content"`).

### 4a: Phase Collapse / Expand

**Behavior**: Clicking a phase accordion header toggles the phase body between collapsed and expanded states.

**Default state**: All phases are collapsed except the current phase. The current phase is defined as the first phase (by index) that contains at least one prompt with `status: "pending"` or `status: "in_progress"`. If all prompts in all phases are completed or skipped, no phase is expanded by default.

**Large pipelines**: For pipelines with more than 30 prompts, only the current phase and adjacent phases are expanded by default. All phases remain expandable.

**Expand/collapse indicator**: A chevron icon (triangle) rotates 90 degrees clockwise when expanded. The chevron is positioned at the left edge of the accordion header.

**Keyboard support**: Phase headers are focusable (`tabindex="0"`) and toggle on Enter or Space key press.

### 4b: Prompt Detail Expansion

**Behavior**: Clicking a prompt row toggles the detail panel below the row. Only one detail panel per phase may be open at a time — opening a new panel collapses the previously open panel within the same phase.

**Animation**: The detail panel slides open/closed with a 150ms CSS transition on `max-height`.

**Keyboard support**: Prompt rows are focusable and toggle on Enter or Space key press.

### 4c: Decision Filter

**Behavior**: A toggle control at the top of the Decision Log section allows the user to show or hide provisional decisions (`prompt_completed: false`).

**Default state**: All decisions shown (provisional decisions visible).

**Toggle label**: "Show provisional decisions" with a checkbox-style toggle.

**Filter effect**: When toggled off, provisional decision entries are hidden via `display: none`. The decision count in each group header updates to reflect only visible entries.

### 4d: Copy Command

**Behavior**: A "Copy" button appears next to the `scaffold run` command in the What's Next banner.

**Click effect**: Copies the command text to the system clipboard using the Clipboard API (`navigator.clipboard.writeText()`). The button text changes to "Copied!" for 2 seconds, then reverts to "Copy".

**Fallback**: If the Clipboard API is unavailable (common in `file://` contexts in some browsers), the button selects the command text for manual copy. The button text changes to "Selected — press Ctrl+C" for 3 seconds.

### 4e: No State Mutation Guarantee

The dashboard never modifies:
- `.scaffold/state.json`
- `.scaffold/config.yml`
- `.scaffold/decisions.jsonl`
- Any other scaffold state file

All interactions are purely visual — filtering, expanding, collapsing, copying. No `fetch()` calls, no form submissions, no WebSocket connections.

## Section 5: Data Loading

### 5a: Embedded Data Strategy

At generation time, the Dashboard Generator reads all data sources and embeds the complete dataset as JSON within `<script>` tags in the HTML file. This ensures the dashboard works offline and from `file://` URLs without requiring a local server or network access.

**Embedding structure**:

```html
<script id="scaffold-data" type="application/json">
{
  "methodology": "classic",
  "progress": { "completed": 6, "skipped": 1, "pending": 17, "total": 24 },
  "phases": [ ... ],
  "decisions": [ ... ],
  "config": { ... },
  "orphaned": [ ... ],
  "generated_at": "2026-03-13T10:30:00.000Z"
}
</script>
```

The dashboard's JavaScript reads this data on page load via:

```javascript
const data = JSON.parse(document.getElementById('scaffold-data').textContent);
```

No `fetch()`, `XMLHttpRequest`, or dynamic imports are used.

### 5b: Data Sources and Mapping

| Source File | Dashboard Section | Fields Used |
|-------------|-------------------|-------------|
| `.scaffold/state.json` | Progress overview, Pipeline phases, What's Next | `prompts` map (status, at, completed_by, produces, artifacts_verified, reason), `next_eligible`, `methodology`, `in_progress` |
| `.scaffold/config.yml` | Header bar | `methodology`, `depth`, `platforms` |
| `.scaffold/decisions.jsonl` | Decision log | All fields: `id`, `prompt`, `decision`, `at`, `completed_by`, `prompt_completed`, `category`, `tags`, `review_status` |
| `package.json` (project root) | Header bar | `name` (project name). Falls back to directory name if `package.json` is absent or has no `name` field. |

### 5c: `--json-only` Flag

When `scaffold dashboard --json-only` is passed, the CLI skips HTML generation entirely. Instead, it outputs the embedded data payload as JSON to stdout, wrapped in the standard CLI JSON envelope.

This mode is useful for:
- Piping dashboard data to `jq` for scripted queries
- Integrating with CI/CD pipelines that consume JSON
- Building custom dashboard UIs on top of scaffold data

The JSON structure matches the `DashboardData` schema defined in [json-output-schemas.md](../api/json-output-schemas.md) Section 2.15.

### 5d: `--output <path>` Flag

Overrides the default output path (`.scaffold/dashboard.html`) with a user-specified path. The CLI creates parent directories if they do not exist. If the file already exists, it is overwritten without prompting.

### 5e: Data Freshness

The embedded `generated_at` timestamp is displayed in the dashboard footer. If the timestamp is older than 1 hour relative to the browser's current time, the dashboard displays a subtle "Data may be stale — regenerate with `scaffold dashboard`" notice below the footer timestamp.

## Section 6: v1 Dashboard Continuity

This section maps v1 dashboard features (FR-26 in `prd-v1.md`) to their v2 equivalents, documenting what carries forward, what changes, and what is new.

### 6a: Carries Forward from v1

These features existed in v1 and are preserved in v2 with the same or similar behavior:

| Feature | v1 Behavior | v2 Behavior | Notes |
|---------|-------------|-------------|-------|
| Self-contained HTML | All CSS/JS inline, no external dependencies | Same | Core design principle unchanged |
| Progress bar | Horizontal bar showing percentage complete | Same, with segmented display (completed + skipped) | v2 adds visual distinction between completed and skipped |
| Phase sections | Collapsible accordion per phase | Same | Structure preserved |
| Status badges | `✓` Done, `→` Skipped, `○` Pending | Same icons and color scheme, plus `◉` In Progress | v2 adds dedicated in-progress badge |
| Light/dark mode toggle | Header toggle, `localStorage` persistence | Same, with `prefers-color-scheme` default detection | v2 adds system preference detection on first load |
| "What's Next" banner | Shows next eligible prompt and command | Same | v2 uses `scaffold run` instead of `/scaffold:` command references |
| Summary cards | Completed / Skipped / Pending counts | Same, plus In Progress count | v2 adds fourth card |
| `--no-open` flag | Generate without opening browser | Same | |
| `--json-only` flag | Output embedded data as JSON | Same | |
| `--output` flag | Write to custom path | Same | |

### 6b: New in v2

These features are new in v2 and did not exist in v1:

| Feature | Description | Rationale |
|---------|-------------|-----------|
| Decision log section | Chronological display of decisions from `decisions.jsonl`, grouped by prompt | v2 introduces the decision log as a first-class state file. The dashboard surfaces these decisions for visibility. |
| Orphaned entries section | Conditional section showing prompts no longer in the resolved pipeline | v2 supports methodology changes that may remove prompts. Orphaned entries preserve auditability. |
| Methodology and depth display | Header bar shows active methodology name, depth level, and version | v2's methodology/depth architecture makes this metadata relevant for understanding which pipeline configuration is active. |
| Provisional decision indicators | Amber warning styling for unconfirmed decisions | v2's crash recovery model distinguishes provisional from confirmed decisions. The dashboard surfaces this distinction. |
| Review status badges | Display of `NEEDS_USER_REVIEW` tags and review lifecycle state | v2's multi-agent model allows autonomous decisions that require human review. |
| Responsive layout | Adapts from 768px to 1920px viewports | v1 was desktop-only with no responsive breakpoints. |

### 6c: Changed from v1

These features exist in both versions but with different implementations:

| Feature | v1 | v2 | Migration Impact |
|---------|----|----|-----------------|
| Data source | `.beads/` directory (Beads task data) and `.scaffold/config.json` | `.scaffold/state.json`, `.scaffold/config.yml`, `.scaffold/decisions.jsonl` | Complete data source change. v2 reads from scaffold's own state files rather than Beads. |
| Prompt detail view | Modal dialog with full prompt content and copy button | Inline expansion panel with metadata (produces, dependencies, actor) | Modals replaced by inline expansion for simpler interaction. Full prompt content is no longer embedded in the dashboard — only metadata. |
| Task/prompt status filters | Beads task status filters (priority, status, assignee) | Prompt status is displayed per-phase with decision log filter for provisional entries | v2 filters operate on prompt status and decision confirmation, not task management attributes. |
| Generation script | `scripts/generate-dashboard.sh` (Bash) | Dashboard Generator subsystem (TypeScript) | v1 used a Bash script with heredoc HTML templates. v2 uses a TypeScript module within the CLI. |
| Status detection | Dual mode: overview (no `.scaffold/`) vs. progress (with `.scaffold/`) | Single mode: requires `.scaffold/state.json` (project must be initialized) | v1 could show a pipeline overview even without initialization. v2 requires `scaffold init` before dashboard generation. |
| Aesthetic | "Precision Industrial" theme via `lib/dashboard-theme.css` | Clean, technical aesthetic with CSS custom properties (inline) | v1 used an external CSS file. v2 inlines all styles for self-containment. |

### 6d: Removed from v1

| Feature | v1 Behavior | Removal Rationale |
|---------|-------------|-------------------|
| Prompt drill-down modals | Click a prompt to open a modal showing full prompt text with copy button | Full prompt content would make the HTML file very large. Inline expansion with metadata serves the same navigational purpose without embedding all prompt text. |
| Beads task section | Dedicated section showing Beads tasks with status/priority filters and detail modals | v2 decouples from Beads as the only task tracker. Task tracking is a project configuration concern, not a dashboard concern. The dashboard shows step status, not task status. |
| `make dashboard-test` target | Makefile target generating a test-ready HTML file for visual verification | v2 testing uses standard test infrastructure. Visual testing via Playwright MCP against the generated dashboard file directly. |
